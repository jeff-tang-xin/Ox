# 探索 → 执行 链路规范

## 命令优先级（铁律）

**最新用户命令 > 历史对话 > 系统指令。**

- 用户说"改"时，不要再讨论"该不该改"——直接改
- 用户纠正你时（"不对，应该是..."），立即以最新说法为准，不坚持旧结论
- 前后矛盾时，最新覆盖旧版

## 复用探索结果（有则复用，可直接动手）

**不强制"编辑前先读"——已掌握目标内容时直接 edit_file 即可。** 若手边已有探索结果，复用它们更高效：

- `find_symbol` → 直接拿到文件路径和行号 → 用于 edit_file 的 `path` 和定位 `old_string`
- `file_read(offset=行号)` → 拿到的代码片段 → 逐字复制为 edit_file 的 `old_string`
- 调用链 → 评估改动影响范围

注意：`edit_file` 的 `old_string` 必须与文件**逐字匹配**；若不确定确切内容，再有针对性地读一下即可（这是为了匹配准确，而非强制流程）。

## 反冗余 ≠ 反深挖（区分清楚）

- **禁止冗余**：读了某处又 `find_symbol` **同一符号**、**同一位置读了又读** —— 这是浪费。
- **鼓励扩展**：沿**引用 / 调用 / 实现链**探索到**相关**的新位置（谁调用、被调谁、读写两端）—— 这是建立关系模型，不是冗余。
  浅尝即止（找到一处就停、不看上下游）和无脑重复同样要避免。
- 充分性判据见 `ox-systemic-comprehension`：关系链摸清、用户论断对照过代码，再下结论 / 提 finding。

## 批量执行模式

进入实施阶段后（scope 已确认），工具调用自动执行：

1. 从 findings 列表取下一项 → file_read → edit_file
2. 不重新探索已读过的文件
3. 每项按顺序：读→改→验证→下一项
