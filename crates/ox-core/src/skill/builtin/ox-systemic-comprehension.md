# 系统性理解 — 探索阶段的工程思维

探索不是"找到要改的那一行就停"。探索的目标是**建立关系模型**：把用户提到的东西在代码里**真正搞清楚**，再下结论。判断"探索够了"的标准不是"找到了"，而是"能讲清楚它怎么连起来"。

> 用户对代码的描述是**待验证的假设，不是事实**。先去代码里确认/证伪，再回应；结论带 `file:line`。

## 五个标准动作（拿到问题就跑这套）

围绕用户提到的每个关键名词（字段 / 方法 / 概念 / 模块）：

1. **概念定位 — "它是什么"**
   `find_symbol(name=…)` 找定义 → `file_read` 看声明/类型/默认值。
   例：用户说"sku 字段" → 找到 `sku` 的声明在哪个类/结构、什么类型、怎么初始化。

2. **读写两端 — "它从哪来、到哪去"（数据流）**
   `code_search` 找**谁写它**（赋值/构造/入库）、**谁读它**（取值/判断/出参）。
   一个字段只看声明等于没看——必须知道它在哪被填充、在哪被消费。

3. **关系链扩展 — "它属于哪个流程、和谁协作"**
   从定义点沿**调用链上下游各走 1–2 跳**：谁调用了这段逻辑、这段逻辑又调用了谁、同一流程里的兄弟方法是哪些。
   目标是看清这块代码的**结构与边界**，不是只盯一个方法。

4. **与现逻辑对齐 — "用户说的在代码里成立吗"**
   把用户论断逐条对照真实代码：成立 / 不成立 / 部分成立，各给出证据行号。
   不附和、不脑补；代码和用户说法冲突时，以代码为准并指出冲突点。

5. **影响面 — "动它会波及哪里"**
   改动前，列出依赖这段逻辑的调用方 / 测试 / 配置，评估波及范围。

针对常见诉求：
- "**这个字段在代码里做什么、和当前逻辑什么关系**" = 动作 1 + 2 + 4。
- "**主动探索所有相关逻辑、了解整个相关结构**" = 动作 3 + 5。

## 探索充分性自检（提 finding / 下结论前过一遍）

- [ ] 关键概念都定位到定义了吗？（动作 1）
- [ ] 读写两端都找到了吗，数据流讲得清吗？（动作 2）
- [ ] 沿关系链走过上下游、看清所属流程了吗？（动作 3）
- [ ] 用户每条论断都对照过真实代码、能给行号了吗？（动作 4）
- [ ] 改动影响面评估过了吗？（动作 5）

五条都能回答 = 理解到位，可以下结论 / 提 finding。还答不上 = **继续探索**，别急着交付。

## 反"浅尝即止"，也反"无脑全读"

- **要深挖**：沿引用 / 调用 / 实现链扩展到**相关**位置，把关系摸清。
- **不铺张**：按"上下游各 1–2 跳 + 真正相关的文件"为界；不要把整个仓库读一遍。
- 优先用**符号/搜索**定位关系（find_symbol / code_search），再用 file_read 看具体实现，避免大段盲读。
