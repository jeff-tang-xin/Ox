# 内容验证逻辑统一化报告

## 📊 优化概览

**优化日期**: 2026-05-06  
**优化目标**: 统一 `file_write` 和 `file_patch` 的内容验证逻辑,消除代码重复  
**测试结果**: ✅ 118/120 测试通过 (2 ignored)  
**状态**: ✅ 已完成

---

## 🎯 问题分析

### 之前的问题

1. **代码重复**: `file_write.rs` 和 `file_patch.rs` 都有各自的 `validate_content()` 函数
2. **逻辑不一致**: 
   - `file_write`: 阈值 10%,只检测控制字符
   - `file_patch`: 阈值 2%,包含复杂的乱码模式检测
3. **维护困难**: 修改验证规则需要同步两个文件
4. **用户体验不一致**: 同样的内容,一个工具接受,另一个拒绝

### 实际案例

```
用户尝试使用 file_patch 替换代码:
"Suspicious Content: 13.3% of replacement content contains non-printable characters"

原因: LLM 生成的代码包含反引号和转义序列,被误判为非打印字符
阈值过严 (2%) 导致正常代码被拒绝
```

---

## ✅ 解决方案

### 1. 创建统一的验证模块

**新文件**: `crates/ox-core/src/tools/content_validation.rs`

**核心函数**:
```rust
pub fn validate_content(content: &str) -> Result<(), String>
```

**验证规则** (与 `file_write` Phase 2 保持一致):
1. ✅ **Null bytes** - 总是拒绝 (确定是损坏)
2. ⚠️ **Replacement characters (U+FFFD)** - >5% 且 >10 个时警告(不阻止)
3. ✅ **Control characters** - >10% 时拒绝 (放宽自 2%)

## 🔍 技术细节

### 验证流程

```
Content Input
  ↓
Check 1: Null bytes (\x00)
  ├─ Found → ❌ Reject immediately
  └─ Not found → Continue
  ↓
Check 2: Replacement chars (U+FFFD)
  ├─ Ratio >5% AND count >10 → ⚠️ Warn (log only)
  └─ Otherwise → Continue
  ↓
Check 3: Control characters (<0x20)
  ├─ Total chars ≤100 → Skip (too short to judge)
  ├─ Ratio >10% → ❌ Reject
  └─ Otherwise → ✅ Allow
  ↓
Return Ok(())
```

### 为什么选择这些阈值?

#### Null bytes: 0% (总是拒绝)
- **理由**: 文本文件中绝不应该出现 null bytes
- **来源**: 二进制数据混入或严重编码错误

#### Replacement chars: 5% + 10 个
- **理由**: 
  - <5% 可能是特殊符号或 emoji
  - >5% 且 >10 个可能真的有编码问题
  - 但不阻止,只是警告 (LLM 可能生成特殊内容)

#### Control characters: 10%
- **理由**:
  - 正常代码可能有少量控制字符 (转义序列)
  - >10% 几乎肯定是损坏或二进制数据
  - 平衡安全性和可用性
