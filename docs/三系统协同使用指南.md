# 三系统协同使用指南

## 概述

本文档展示了 Ox 项目中**文件索引系统**、**记忆系统**和**上下文系统**如何协同工作，提供智能的项目探索和代码理解能力。

---

## 核心功能

### 1. 文件索引系统 - 层级目录导航

#### 问题
传统 `file_list` 工具在大型项目（3万+文件）中会返回所有文件列表，导致上下文窗口爆炸。

#### 解决方案
使用层级目录导航，LLM 可以逐层探索项目结构。

#### 使用示例

```rust
// 在 ToolContext 中调用
let listing = ctx.file_index.list_directory("src/agent", false)?;

// 输出格式：
// 📁 Directory: src/agent
// Total files: 156
//
// 📂 Subdirectories:
//   - workflow/ (45 files)
//   - session/ (32 files)
//   - mode/ (28 files)
//
// 📄 Files:
//   [123] mod.rs (.rs)
//   [124] engine.rs (.rs)
//   [125] interrupt.rs (.rs)
//
// 💡 Tip: Use 'path' parameter to explore subdirectories (e.g., 'src/agent/workflow/')
```

#### LLM 交互流程

```
User: "帮我分析一下这个项目的结构"

Assistant: [调用 file_list 无参数]
→ 看到顶层目录：src/, docs/, crates/

Assistant: [调用 file_list path="src/"]
→ 看到 src/ 下的子目录：agent/, tools/, memory/

Assistant: [调用 file_list path="src/agent/"]
→ 看到 agent/ 下的模块：workflow/, session/, mode/

Assistant: [调用 file_read path="src/agent/mod.rs"]
→ 读取入口文件，理解整体架构
```

### 2. 记忆系统 - 文件关联检索

#### 问题
传统记忆检索只基于语义相似度，无法感知用户当前正在工作的文件。

#### 解决方案
为记忆节点添加 `related_files` 字段，实现基于文件的精准检索。

#### 自动关联机制

当 LLM 调用工具时，系统自动提取文件路径并关联到记忆：

```rust
// LLM 调用 file_read
{
  "tool": "file_read",
  "arguments": {"path": "src/auth.rs"}
}

// 系统自动创建记忆并关联文件
MemoryNode {
  content: "Read file: src/auth.rs",
  node_type: MemoryNodeType::Fact,
  related_files: vec!["src/auth.rs".to_string()],
  // ...
}
```

#### 手动关联 API

```rust
// 创建记忆时关联文件
let memory = MemoryNode::new(
    "Authentication uses JWT tokens with 24h expiry".to_string(),
    MemoryNodeType::Architectural,
    Some(project_id),
    "rust".to_string(),
    MemorySource::LlmExtraction,
)
.with_related_file("src/auth.rs")
.with_related_file("src/middleware/auth.rs");

memory_manager.store(memory);
```

#### 基于文件的检索

```rust
// 检索与特定文件相关的记忆
let related_memories = memory_manager.retrieve_by_files(
    &["src/auth.rs".to_string()],
    &Some("my-project"),
    5  // limit
)?;

// 返回的记忆包含：
// 1. 直接关联 src/auth.rs 的记忆（权重 1.5x）
// 2. 语义相关且有关联文件的记忆（权重 1.0x）
```

### 3. 上下文系统 - 智能意图识别

#### 问题
固定的 token 预算分配无法适应不同的使用场景。

#### 解决方案
检测用户意图，动态调整 token 预算分配。

#### 意图类型

| 意图类型 | 典型查询 | 预算策略 |
|---------|---------|---------|
| **Exploration** | "show me the project", "what files are there" | history: 15%, memory: 1% |
| **CodeUnderstanding** | "how does auth work", "explain this function" | history: 8%, memory: 5% |
| **CodeModification** | "add a feature", "fix this bug" | history: 8%, memory: 5% |
| **General** | 其他对话 | history: 10%, memory: 2% (默认) |

#### 使用示例

```rust
use ox_core::context::{detect_intent, extract_recent_files};

// 1. 检测用户意图
let user_query = "How does the authentication system work?";
let intent = detect_intent(user_query);
// → UserIntent::CodeUnderstanding

// 2. 根据意图调整预算
let budgets = context_builder.budgets_for_intent(128_000, intent);
// → memory: 6400 tokens (5%), history: 10240 tokens (8%)

// 3. 提取最近访问的文件
let recent_files = extract_recent_files(&history, 5);
// → ["src/auth.rs", "src/middleware/mod.rs"]

// 4. 检索相关文件记忆
let file_memories = memory_manager.retrieve_by_files(
    &recent_files,
    &Some(project_id),
    3
)?;

// 5. 结合语义检索
let semantic_memories = memory_manager.retrieve_with_rerank(
    user_query,
    &Some(project_id),
    5,
    Some(&embedder),
    None
)?;

// 6. 合并记忆并去重
let mut all_memories = file_memories;
all_memories.extend(semantic_memories);
all_memories.dedup_by_key(|m| m.id.clone());

// 7. 格式化并注入上下文
let memory_context = memory_manager.format_memory_context(&all_memories, true);
let messages = context_builder.build(
    &system_prompt,
    &memory_context,
    &history,
    128_000
);
```

## 完整工作流示例

### 场景：用户询问认证逻辑

```
User: "How does authentication work in this project?"

Step 1: Context System detects intent
  → detect_intent() returns UserIntent::CodeUnderstanding
  → Adjust budgets: memory 5%, history 8%

Step 2: Extract recent files from history
  → extract_recent_files() finds ["src/main.rs", "src/lib.rs"]

Step 3: File-based memory retrieval
  → retrieve_by_files(["src/main.rs", "src/lib.rs"]) 
  → Returns 2 memories about project structure

Step 4: Semantic memory retrieval
  → retrieve_with_rerank("authentication", ...)
  → Returns 3 memories about JWT, middleware, auth.rs

Step 5: File index navigation (if needed)
  → If memories mention "src/auth.rs", LLM can call:
  → file_list(path="src/") to locate the file
  → file_read(path="src/auth.rs") to read implementation

Step 6: Build final context
  → Merge system prompt + memories + recent history
  → Send to LLM with optimized token budget
```

## 最佳实践

### 1. 文件探索
- ✅ 使用 `file_list` 无参数查看顶层结构
- ✅ 逐层深入：`src/` → `src/agent/` → `src/agent/workflow/`
- ❌ 避免一次性列出所有文件

### 2. 记忆管理
- ✅ 让系统自动关联文件（无需手动操作）
- ✅ 使用 `retrieve_by_files()` 获取上下文相关记忆
- ✅ 结合语义检索和文件检索
- ❌ 不要过度依赖单一检索方式

### 3. 上下文优化
- ✅ 根据用户意图动态调整预算
- ✅ 优先检索最近访问文件的相关记忆
- ✅ 限制单次检索结果数量（5-10 条）
- ❌ 避免在上下文中注入过多记忆

## 故障排查

### 问题：记忆没有自动关联文件

**原因**：工具调用的 JSON 格式不正确

**解决**：确保工具参数包含 `path` 字段
```json
// ✅ 正确
{"path": "src/auth.rs"}

// ❌ 错误
{"file": "src/auth.rs"}
```

### 问题：意图识别不准确

**原因**：关键词匹配过于简单

**解决**：扩展 `detect_intent()` 中的关键词列表
```rust
let exploration_keywords = [
    // 添加更多同义词
    "show me", "list", "what files", "project structure",
    "directory", "explore", "browse", "overview", "structure",
    "tree", "layout", "organization"  // 新增
];
```

### 问题：文件检索返回空结果

**原因**：记忆中未设置 `related_files`

**解决**：检查数据库 schema 是否包含 `related_files` 列
```sql
-- 验证列存在
PRAGMA table_info(memories);

-- 如果缺失，手动添加（仅旧数据库需要）
ALTER TABLE memories ADD COLUMN related_files TEXT NOT NULL DEFAULT '[]';
```
