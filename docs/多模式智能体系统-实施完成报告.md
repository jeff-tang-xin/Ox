## 🎯 架构设计亮点

### 1. 注册化设计

```rust
// 工作流可注册、可扩展
pub struct WorkflowEngine {
    workflows: HashMap<String, Workflow>,  // 已注册的工作流
    current_workflow: Option<Workflow>,    // 当前激活的工作流
}

// 新模式只需注册 Workflow 即可
engine.register_workflow(create_custom_workflow());
```

### 2. 强制性工作流

**不是依赖 LLM 自觉遵守规则，而是代码强制执行：**

```rust
// 检查当前步骤是否允许工具调用
if !workflow_engine.allows_tool_execution() {
    return Err("Tools not allowed in current step");
}

// 检查是否需要用户确认
if workflow_engine.requires_user_confirmation() {
    intervention_manager.queue_intervention(...);
    return Ok(()); // 等待用户响应
}

// 验证步骤完成状态
if !workflow_engine.validate_current_step() {
    return Err("Current step not completed");
}
```

### 3. Session 状态隔离

```rust
pub struct SessionState {
    pub session_id: String,
    pub current_mode: String,              // free/spec/council
    pub current_workflow: String,          // 工作流 ID
    pub current_step_index: usize,         // 当前步骤
    pub variables: HashMap<String, String>, // Session 变量
}

// 支持 Session 切换
state_registry.switch_session("session_2");
```

### 4. 人工介入机制

```rust
// Step 3: Await Spec Confirmation
WorkflowStep::new("await_spec_confirmation", "...")
    .require_confirmation()      // ⚠️ 必须用户确认
    .disallow_tools()            // ❌ 禁止工具调用
```

## 📊 三种模式的工作流

### Spec Mode（6步强制流程）

```
Step 1: Requirement Analysis
  ↓ 自动（LLM 分析任务类型）
Step 2: Generate spec.md
  ↓ 代码验证（检查文件是否存在）
Step 3: Await Spec Confirmation ⚠️ USER CONFIRMATION REQUIRED
  ↓ 用户点击"确认"后继续
Step 4: Generate task.md
  ↓ 代码验证（检查文件是否存在）
Step 5: Await Task Confirmation ⚠️ USER CONFIRMATION REQUIRED
  ↓ 用户点击"确认"后继续
Step 6: Execute Code ✅ TOOLS ALLOWED
  ↓ LLM 执行实际代码修改
```

**关键约束：**
- Step 3 和 Step 5 **必须**等待用户确认（代码强制）
- Step 2 和 Step 4 通过验证器检查文件是否存在
- Step 6 之前**禁止**工具调用（代码强制）

### Council Mode（6步辩论流程）

```
Step 1: Topic Definition ⚠️ USER CONFIRMATION
  ↓
Step 2: Proposal Phase ❌ NO TOOLS
  ↓
Step 3: Review Phase ❌ NO TOOLS
  ↓
Step 4: Rebuttal Phase ❌ NO TOOLS
  ↓
Step 5: Arbitration ❌ NO TOOLS
  ↓
Step 6: Conclusion ⚠️ USER CONFIRMATION
```

**关键约束：**
- 全程禁用工具调用（内部对话）
- Step 1 和 Step 6 需要用户确认

### Free Mode（单步无限制）

```
Step 1: Free Interaction ✅ FULL ACCESS
```

## 🔧 技术实现细节

### 1. 向后兼容性

为了最小化对现有代码的影响，保留了旧字段但标记为 deprecated：

```rust
#[deprecated(note = "Use workflow_state instead")]
pub spec_content: String,

#[deprecated(note = "Use workflow_state instead")]
pub spec_active: bool,
```

这样可以：
- ✅ 现有代码无需立即修改
- ✅ 编译器会发出警告提醒迁移
- ✅ 逐步迁移到新架构

### 2. 状态机驱动

所有模式转换通过状态机管理：

```rust
pub enum WorkflowState {
    Free,
    Spec { step: SpecWorkflowStep, spec_content: String },
    Council { step: CouncilWorkflowStep, topic: Option<String> },
}
```

### 3. 验证器系统

```rust
// 注册验证器
state_registry.register_validator("check_spec_file_exists", |state| {
    state.has_variable("spec_file_created")
});

// 在 WorkflowStep 中引用
WorkflowStep::new("generate_spec", "...")
    .with_validator("check_spec_file_exists")
```

## 🎓 使用示例

### 激活 Spec 模式

```bash
# 用户输入
/spec 实现用户认证功能

# 系统响应
1. 激活 spec_workflow
2. 设置 Session 状态: mode=spec, step=0
3. LLM 收到增强的系统提示词:
   - TASK_TYPE_PROMPT
   - Step 1 的具体指令
4. LLM 分析任务类型并分类
```

### 用户确认流程

```
LLM: 已生成 spec.md，请确认是否继续？

UI 显示:
  ┌─────────────────────────────────┐
  │ Spec Confirmation Required      │
  │                                 │
  │ 已生成规范文件 .ox/spec.md      │
  │ 是否确认并继续生成任务计划？     │
  │                                 │
  │  [✅ 确认]  [❌ 拒绝]  [✏️ 编辑] │
  └─────────────────────────────────┘

用户点击: ✅ 确认

系统:
  1. InterventionManager 接收 Confirmed 响应
  2. 返回 InterventionAction::Proceed
  3. WorkflowEngine 推进到 Step 4
  4. 设置 session.variables["spec_confirmed"] = "true"
  5. LLM 继续生成 task.md
```

## 🔍 关键技术点

1. **状态机驱动** - 所有模式转换通过状态机管理
2. **代码级强制** - 不是依赖 LLM 遵守规则，而是代码强制执行
3. **可插拔设计** - 新模式只需注册 Workflow 即可
4. **Session 隔离** - 每个 Session 独立状态，切换不丢失
5. **人工介入优先** - 关键步骤必须用户确认
6. **向后兼容** - 保留旧 API，逐步迁移
