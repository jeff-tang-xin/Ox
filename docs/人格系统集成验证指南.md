# 人格系统集成验证指南

## 🚀 快速开始

### 1. 编译和运行

```bash
# 确保编译成功
cargo build --package ox-cli

# 运行程序
cargo run --package ox-cli
```

### 2. 观察启动日志

启动后，在 `~/.ox/logs/ox.log` 中应该看到：

```
[INFO] Embedding model loaded: ... (如果启用了 embedding)
[WARN] Failed to load EMA history: ... (首次运行正常，数据库为空)
```

## 🧪 测试场景

### 场景 1: 文件写入注册

**步骤**:
1. 启动 Ox CLI
2. 输入: `/write test.txt Hello World`
3. 查看日志: `tail -f ~/.ox/logs/ox.log | grep "IMPLICIT FEEDBACK"`

**预期输出**:
```
[DEBUG] [IMPLICIT FEEDBACK] Registered write: "/path/to/test.txt", total: 1
```

**验证**:
- ✅ 日志中出现 "Registered write"
- ✅ `total` 计数器递增

---

### 场景 2: 改动检测

**步骤**:
1. 用编辑器打开 `test.txt`
2. 大幅修改内容（例如添加 10 行新代码）
3. 保存文件
4. 回到 Ox CLI，输入任意内容（如 "hello"）
5. 查看日志

**预期输出**:
```
[INFO] [IMPLICIT FEEDBACK] Major rewrite: "/path/to/test.txt" (65.3%)
```

或者如果改动很小：
```
[DEBUG] [IMPLICIT FEEDBACK] Accepted: "/path/to/test.txt" (change: 2.1%)
```

**验证**:
- ✅ 检测到改动并记录比例
- ✅ 根据比例分类为 Minor/Major/VeryStrong

---

### 场景 3: EMA 更新

**步骤**:
1. 重复场景 1 和 2 多次（至少 10 次文件写入）
2. 查看日志中的 "FEEDBACK METRICS"

**预期输出**:
```
[DEBUG] [FEEDBACK METRICS] accept_rate=0.75, total=10, accepted=8
```

**验证数据库**:
```bash
sqlite3 ~/.ox/db/memories_overall.db
SELECT * FROM ema_trends WHERE metric_name = 'code_accept_rate';
```

应该看到类似：
```
code_accept_rate|0.75|0.02|10|1714464000
```

---

### 场景 4: 显式反馈

**步骤**:
1. 输入: `/feedback good`
2. 输入: `/feedback bad`
3. 查看日志

**预期输出**:
```
[INFO] [EXPLICIT FEEDBACK] Good: 1/1
[INFO] [EXPLICIT FEEDBACK] Bad: 1/2
```

**UI 提示**:
```
✅ Feedback noted: positive. Memory reinforced.
⚠️  Feedback noted: negative. Persona adjusted.
```

---

### 场景 5: 满意度评估和回滚

**步骤**:
1. 进行多轮对话，让 AI 生成代码
2. 手动修改生成的代码（模拟不满意）
3. 多次使用 `/feedback bad`
4. 等待对话结束（TurnDone 事件）
5. 查看日志中的 "SATISFACTION EVAL"

**预期输出**:
```
[INFO] [SATISFACTION EVAL] explicit=0.30, tool=0.95, code_accept=0.60, overall=0.52
[WARN] [ROLLBACK NEEDED] Current: 0.52, Baseline: 0.75, Degradation: 0.23
```

如果触发了回滚：
```
[WARN] [ROLLBACK] Snapshot: abc123..., Score: 0.52 → 0.78
```

**UI 提示**:
```
⚠️  Warning: Satisfaction dropped (0.52 < 0.75), but no rollback point available
```
或
```
⚠️  Persona rolled back due to satisfaction drop (0.52 → 0.78)
```

## 🔍 调试技巧

### 1. 实时监控所有反馈事件

```bash
# 在一个终端窗口运行
tail -f ~/.ox/logs/ox.log | grep -E "(IMPLICIT FEEDBACK|SATISFACTION|ROLLBACK|EXPLICIT FEEDBACK)"
```

### 2. 检查数据库状态

```bash
# 查看 EMA 趋势
sqlite3 ~/.ox/db/memories_overall.db <<EOF
.mode column
.headers on
SELECT * FROM ema_trends;
EOF

# 查看人格快照
sqlite3 ~/.ox/db/memories_overall.db <<EOF
.mode column
.headers on
SELECT snapshot_id, language, satisfaction_score, created_at 
FROM persona_snapshots 
ORDER BY created_at DESC 
LIMIT 5;
EOF
```

### 3. 强制触发回滚测试

```bash
# 在 Ox CLI 中连续发送负面反馈
/feedback bad
/feedback bad
/feedback bad
/feedback bad
/feedback bad

# 然后进行一轮对话，观察是否触发回滚
```

### 4. 查看当前计数器状态

在代码中添加临时调试输出（可选）：

```rust
// 在 TurnDone 处理中添加
tracing::info!(
    "[COUNTERS] total={}, accepted={}, explicit={}, good={}",
    app.total_file_writes,
    app.accepted_file_writes,
    app.explicit_feedback_count,
    app.good_feedback_count
);
```

## ❓ 常见问题

### Q1: 为什么没有看到 "Registered write" 日志？

**可能原因**:
1. 文件写入失败（is_error = true）
2. 输出格式不匹配（期望 "Wrote N bytes to /path"）
3. 无法从消息中提取内容

**解决方法**:
```bash
# 检查是否有 file_write 错误
tail -f ~/.ox/logs/ox.log | grep "file_write"

# 确认工具调用成功
tail -f ~/.ox/logs/ox.log | grep "ToolResult"
```

### Q2: EMA 加载失败怎么办？

**正常情况**: 首次运行时数据库为空，会看到警告但继续运行。

**如果不是首次**:
```sql
-- 检查表是否存在
sqlite3 ~/.ox/db/memories_overall.db ".tables"

-- 应该看到: ema_trends  persona_snapshots

-- 检查数据
SELECT * FROM ema_trends;
```

### Q3: 如何禁用隐式反馈？

**方法 1**: 在代码中条件判断（需要重新编译）
```rust
// 在所有隐式反馈代码前添加
if config.feedback.enabled {
    // ... feedback logic ...
}
```

**方法 2**: 设置 `config.persona.auto_evolve = false`（禁用自动演化，间接禁用回滚）

### Q4: 回滚阈值太高/太低怎么办？

当前硬编码为 0.2（20% 下降）。

**修改位置**: `crates/ox-core/src/feedback/rollback.rs:135`

```rust
let degradation = baseline_satisfaction - current_satisfaction;
if degradation > 0.2 {  // ← 修改这个值
    // ... rollback logic ...
}
```

建议范围: 0.15 - 0.30

## 📊 性能监控

### CPU 使用率

```bash
# Linux/macOS
top -p $(pgrep ox)

# Windows PowerShell
Get-Process ox | Select-Object CPU,WorkingSet
```

**预期**: < 5% CPU（空闲时），< 15%（活跃使用时）

### 内存使用

```bash
# 检查 SQLite 文件大小
ls -lh ~/.ox/db/memories_overall.db

# 预期: < 1MB（即使有数百条记录）
```

### I/O 延迟

在日志中搜索持久化时间：
```bash
grep "persist_to_store" ~/.ox/logs/ox.log

# 应该看到异步执行，不阻塞主线程
```
