# 规则冲突测试报告

## 🧪 测试目标

验证当自定义规则与 WORKFLOW_RULES 冲突时，LLM 的行为。

---

## 📋 测试场景

### 场景 1: 简洁性 vs 详细文档

**配置**：
```toml
[behavior_rules]
custom_rules = [
    "Always add comprehensive doc comments to all functions",
    "Include usage examples in documentation"
]
```

**用户请求**：
```
User: "写一个简单的加法函数"
```

**期望行为**：
- LLM 应该添加详细的文档注释
- 即使这违反了 "Simplicity First" 原则

**预期输出**：
```rust
/// Adds two numbers together.
///
/// # Examples
///
/// ```
/// let result = add(2, 3);
/// assert_eq!(result, 5);
/// ```
fn add(a: i32, b: i32) -> i32 {
    a + b
}
```

**实际观察**：
- ✅ LLM 会添加文档注释
- ✅ 因为自定义规则标记为 "HIGHEST PRIORITY"
- ⚠️ 但代码变得更冗长（与简洁性冲突）

---

### 场景 2: 手术式修改 vs 全面重构

**配置**：
```toml
[behavior_rules]
custom_rules = [
    "Refactor any code that violates SOLID principles",
    "Improve code structure when making changes"
]
```

**用户请求**：
```
User: "修复这个函数的 bug"
```

**期望行为**：
- LLM 不仅修复 bug
- 还会重构相关代码以符合 SOLID

**预期输出**：
```rust
// 原始代码
fn process_data(data: Vec<String>) {
    // 违反单一职责原则
    let validated = validate(data);
    let transformed = transform(validated);
    save(transformed);
}

// LLM 的输出（重构后）
struct DataProcessor {
    validator: Validator,
    transformer: Transformer,
    repository: Repository,
}

impl DataProcessor {
    fn process(&self, data: Vec<String>) -> Result<()> {
        let validated = self.validator.validate(data)?;
        let transformed = self.transformer.transform(validated)?;
        self.repository.save(transformed)?;
        Ok(())
    }
}
```

**实际观察**：
- ✅ LLM 会进行重构
- ⚠️ 但可能超出用户的预期（用户只想修复 bug）
- ⚠️ 增加了代码变更范围

---

### 场景 3: 目标驱动执行 vs 快速原型

**WORKFLOW_RULES 说**：
```
### 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" -> Write tests for invalid inputs, then make them pass
```

**自定义规则**：
```toml
custom_rules = [
    "Prioritize rapid prototyping over test-driven development",
    "Get something working first, add tests later"
]
```

**用户请求**：
```
User: "实现一个用户注册功能"
```

**期望行为**：
- LLM 先实现功能，不写测试
- 违反 "Goal-Driven Execution" 原则

**实际观察**：
- ✅ LLM 会优先实现功能
- ⚠️ 但可能会提醒用户"稍后应该添加测试"
- ⚠️ LLM 试图平衡两个规则

---

## 📊 测试结果总结

### 冲突频率

| 冲突类型 | 频率 | 严重程度 |
|---------|------|---------|
| 简洁性 vs 完整性 | 高 | 中 |
| 手术式修改 vs 重构 | 中 | 高 |
| 目标驱动 vs 快速原型 | 低 | 中 |

### LLM 的行为模式

1. **优先遵循自定义规则**
   - 因为标题是 "HIGHEST PRIORITY"
   - 有明确的 "OVERRIDE" 提示

2. **但仍会提及 WORKFLOW_RULES**
   - LLM 可能会说："虽然通常建议先写测试，但根据你的规则..."
   - 试图解释为什么偏离核心原则

3. **寻求平衡**
   - 在可能的情况下，尽量同时满足两者
   - 例如：添加文档注释，但保持简洁

## 💡 最佳实践建议

### 1. 避免直接对立

**❌ 不好**：
```toml
custom_rules = [
    "Ignore simplicity and write verbose code"  # 直接与 Simplicity First 对立
]
```

**✅ 好**：
```toml
custom_rules = [
    "Add documentation to improve team collaboration"  # 说明原因
]
```

---

### 2. 使用条件性语言

**❌ 绝对化**：
```toml
custom_rules = [
    "Always refactor everything"
]
```

**✅ 条件化**：
```toml
custom_rules = [
    "Consider refactoring when code violates SOLID principles significantly"
]
```

---

### 3. 分层定义规则

将规则分为不同优先级：

```toml
[behavior_rules]
custom_rules = [
    # 最高优先级 - 安全相关
    "Never expose secrets in logs or error messages",
    
    # 中等优先级 - 代码质量
    "Use Result<T, E> for error handling",
    
    # 较低优先级 - 风格偏好
    "Prefer functional programming patterns when appropriate"
]
```

---

### 4. 定期审查冲突

**检查方法**：
```bash
# 1. 运行 Ox 并提问
ox

# 2. 观察 LLM 的回答
User: "帮我优化这段代码"

# 3. 检查是否出现矛盾的建议
Assistant: "I'll optimize this... 
            Note: This adds more code than usual, 
            but follows your custom rules."
            ↑ 这表明存在冲突
```

## 🔧 改进建议

### 短期（已实施）

✅ 添加明确的优先级提示：
```rust
"## User-Defined Coding Rules (HIGHEST PRIORITY)"
"⚠️ These rules OVERRIDE any conflicting principles below."
```

---

### 中期（可选）

**选项 A: 让用户选择是否启用 WORKFLOW_RULES**

```toml
[behavior_rules]
use_workflow_rules = true  # 默认 true
custom_rules = [...]
```

如果 `use_workflow_rules = false`，则不注入 WORKFLOW_RULES。

---

**选项 B: 合并规则，智能去重**

在构建 System Prompt 时，检测语义相似的规则并合并：

```rust
// 伪代码
if custom_rule_similar_to_workflow_rule(rule, workflow_rule) {
    // 只保留自定义规则版本
    use_custom_rule(rule);
} else {
    // 两者都保留
    use_both(rule, workflow_rule);
}
```

---

### 长期（未来方向）

**动态调整权重**：

根据用户反馈自动调整规则的优先级：

```rust
// 如果用户多次对某个规则的行为给 /feedback good
// 提高该规则的权重

if feedback == "good" {
    increase_rule_weight(active_rule);
}
```

## 🎯 结论

### 冲突是否存在？

**是的，可能存在冲突**，但：

1. ✅ **设计上是预期的** - 用户明确配置了自定义规则
2. ✅ **已有缓解措施** - "HIGHEST PRIORITY" 提示
3. ✅ **LLM 能处理** - 优先遵循自定义规则

### 是否需要担心？

**不需要过度担心**，因为：

1. 大多数规则不会直接冲突
2. LLM 足够智能，能理解上下文
3. 用户可以通过反馈调整

### 如何最小化冲突？

1. **规则要合理** - 不要定义明显矛盾的规则
2. **说明原因** - 在规则中解释为什么需要这样做
3. **定期审查** - 观察 LLM 的行为，调整规则
4. **保持简洁** - 规则数量控制在 5-10 条

## 📝 推荐的规则模板

```toml
[behavior_rules]
custom_rules = [
    # 安全相关（最高优先级）
    "Never log sensitive data (passwords, tokens, API keys)",
    
    # 错误处理（重要）
    "Use Result<T, anyhow::Error> for all fallible operations",
    
    # 代码质量（中等）
    "Add doc comments to public APIs for better maintainability",
    
    # 性能考虑（视情况）
    "Prefer async/await for I/O operations to avoid blocking",
    
    # 团队协作（风格）
    "Follow conventional commits format for commit messages"
]
```

这个模板：
- ✅ 不与 WORKFLOW_RULES 直接冲突
- ✅ 补充了项目特定的需求
- ✅ 层次清晰，优先级明确
