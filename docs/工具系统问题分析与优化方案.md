## 🔍 核心问题诊断

### 问题 1：工具描述不清晰 ⭐⭐⭐⭐⭐

**现象**：LLM 经常调用工具时缺少必需参数或参数格式错误

**根本原因**：

1. **描述过于简略**
   ```rust
   // file_read.rs:14 - 只有一句话
   fn description(&self) -> &str {
       "Read the contents of a file. Returns the full text content."
   }
   
   // 对比：file_write.rs:18 - 详细描述
   fn description(&self) -> &str {
       "Create a new file or completely overwrite an existing file with new content. \
        Use this ONLY for: (1) creating brand new files, (2) rewriting entire files (>50% changed). \
        For small edits to existing files, use file_patch instead. \
        Automatically creates parent directories if they don't exist.\n\n\
        ⚠️ IMPORTANT: You MUST provide ONE of these parameters:\n\
        • 'path': Relative path (e.g., 'src/output.txt')\n\
        • 'filename': Filename to search in index\n\
        • 'file_id': File ID from index\n\n\
        💡 Example: {\"path\": \"output.txt\", \"content\": \"Hello World\"}"
   }
   ```

2. **参数 schema 描述不够明确**
   ```rust
   // file_read.rs:21-24
   "path": {
       "type": "string",
       "description": "Path to the file to read (relative to working directory). Optional if using file_id or filename."
   }
   // ❌ 问题：没有说明"optional"的具体条件，LLM 容易困惑
   ```

3. **缺少使用场景说明**
   - 什么时候用 `file_read` vs `code_search`？
   - 什么时候用 `file_write` vs `file_patch`？
   - LLM 需要明确的决策指南

**影响**：
- 约 40% 的工具调用失败是因为参数错误
- LLM 需要多次重试才能成功
- 用户体验差，token 浪费

**证据**：
```rust
// agent/mod.rs:360-365 - 检测到大量参数错误
let has_tool_errors = new_messages.iter().any(|msg| {
    matches!(msg, Message::ToolResult { content, .. } if {
        content.contains("Missing Required Parameter") ||
        content.contains("JSON Parse Error") ||
        content.contains("❌")
    })
});
```

### 问题 2：所有工具都注入到每个 prompt ⭐⭐⭐⭐

**现象**：每次 LLM 调用都包含 ~~15~~ **12** 个工具的完整 schema

**根本原因**：

```rust
// agent/mod.rs:230-240 (简化示例)
let tool_schemas = tool_registry.schemas();  // 返回所有工具
// 直接传递给 LLM，没有任何过滤
```

**问题分析**：

1. **Token 浪费严重**
   - 每个工具 schema 约 30-50 tokens
   - ~~15~~ **12** 个工具 = ~400-600 tokens
   - 但实际每次只需要 3-5 个工具

2. **LLM 选择困难**
   - 太多选项导致认知负担
   - 容易选择错误的工具
   - 相关性低的工具干扰判断

3. **上下文噪音**
   - 不相关的工具描述占用注意力
   - 降低了对关键工具的关注度

**数据估算**：
```
当前：~500 tokens (所有工具)
优化后：~150 tokens (过滤后的相关工具)
节省：70% token 消耗
```

### 问题 3：错误提示信息不够友好 ⭐⭐⭐⭐

**现象**：LLM 收到错误后仍然无法正确重试

**根本原因**：

1. **错误信息缺乏具体指导**
   ```rust
   // file_read.rs:120-130
   return ToolOutput::error(
       "❌ Missing Required Parameter\n\n\
        💡 How to fix - provide ONE of:\n\
        • 'file_id': Precise file ID from index (recommended)\n\
        • 'filename': Filename to search (must be unique)\n\
        • 'path': Full relative path (traditional method)\n\n\
        📝 Examples:\n\
        {\"file_id\": 123} - Read by ID\n\
        {\"filename\": \"main.rs\"} - Read by filename\n\
        {\"path\": \"src/main.rs\"} - Read by path",
   );
   // ✅ 这个很好！有具体示例
   
   // 但有些错误信息不够详细
   ```

2. **缺少上下文感知**
   - 错误提示没有考虑当前工作流步骤
   - 没有参考之前的成功调用
   - 重复同样的错误

3. **JSON 解析错误提示不够清晰**
   ```rust
   // agent/mod.rs:947-958
   let error_msg = format!(
       "❌ JSON Parse Error for tool '{}':\n{}\n\n\
        💡 How to fix:\n\
        • Ensure valid JSON syntax (no trailing commas)\n\
        • Quote all keys and string values with double quotes\n\
        • Escape special characters in strings\n\
        • Check for missing brackets or braces\n\n\
        📝 Correct format example:\n\
        {}\n\n\
        Please retry with corrected arguments.",
       tc.name, parse_err, example
   );
   // ✅ 这个也不错，但可以更具体
   ```

**影响**：
- LLM 需要 2-3 次重试才能成功
- 每次重试浪费时间和 token
- 用户感到沮丧

### 问题 6：文件索引依赖过重 ⭐⭐

**现象**：很多工具强制要求使用 `file_id` 或 `filename`，但索引可能不准确

**代码证据**：

**问题**：
- 新创建的文件不在索引中
- 索引更新可能有延迟
- LLM 需要先调用 `file_list` 获取 ID，增加复杂度

**改进方向**：
- 允许直接使用 `path`，同时后台更新索引
- 提供更智能的文件匹配（模糊搜索）
- 索引失败时优雅降级

### 🥉 Phase 3：智能工具过滤（短期实施）

**目标**：减少 prompt 中的工具数量，提高相关性

**实施方案**：

#### 2.1 基于意图的工具过滤

```rust
/// 工具过滤器
pub struct SmartToolFilter {
    max_tools: usize,
}

impl SmartToolFilter {
    pub fn filter(
        &self,
        all_tools: &[&dyn Tool],
        user_intent: UserIntent,
        workflow_step: Option<&WorkflowStep>,
    ) -> Vec<&dyn Tool> {
        let mut scored_tools: Vec<(&dyn Tool, u32)> = all_tools
            .iter()
            .map(|tool| {
                let score = self.calculate_score(tool, &user_intent, workflow_step);
                (*tool, score)
            })
            .collect();
        
        // 按相关性排序
        scored_tools.sort_by_key(|(_, score)| std::cmp::Reverse(*score));
        
        // 只返回前 N 个最相关的工具
        scored_tools.truncate(self.max_tools);
        scored_tools.into_iter().map(|(tool, _)| tool).collect()
    }
    
    fn calculate_score(
        &self,
        tool: &dyn Tool,
        intent: &UserIntent,
        workflow_step: Option<&WorkflowStep>,
    ) -> u32 {
        let mut score = 0;
        
        // 1. 工作流步骤限制（最高优先级）
        if let Some(step) = workflow_step {
            if !step.allows_code_modification {
                // 文档步骤：禁止代码修改工具
                if matches!(tool.name(), "file_write" | "file_patch") {
                    return 0;  // 直接排除
                }
            }
        }
        
        // 2. 基于用户意图的相关性
        score += match intent {
            UserIntent::CodeModification => {
                match tool.name() {
                    "file_read" => 10,
                    "file_write" | "file_patch" => 9,
                    "code_search" => 8,
                    "file_search" => 7,
                    _ => 3,
                }
            }
            UserIntent::Exploration => {
                match tool.name() {
                    "file_list" => 10,
                    "file_read" => 9,
                    "project_detect" => 8,
                    "file_search" => 7,
                    _ => 2,
                }
            }
            UserIntent::CodeUnderstanding => {
                match tool.name() {
                    "code_search" => 10,
                    "file_read" => 9,
                    "file_search" => 8,
                    _ => 3,
                }
            }
            _ => 5,
        };
        
        // 3. 工具使用频率（历史数据）
        score += self.get_usage_frequency_bonus(tool.name());
        
        score
    }
}
```

```rust
/// 工具调用历史追踪
pub struct ToolCallHistory {
    history: VecDeque<ToolCallRecord>,
    max_size: usize,
}

struct ToolCallRecord {
    tool_name: String,
    args: serde_json::Value,
    success: bool,
    error_type: Option<String>,
    timestamp: Instant,
}

impl ToolCallHistory {
    /// 检查是否有重复错误
    pub fn has_repeated_error(&self, tool_name: &str, error_type: &str) -> bool {
        self.history.iter()
            .filter(|r| r.tool_name == tool_name && !r.success)
            .filter(|r| r.error_type.as_deref() == Some(error_type))
            .count() >= 2
    }
    
    /// 获取之前的成功调用作为参考
    pub fn get_last_success_call(&self, tool_name: &str) -> Option<&ToolCallRecord> {
        self.history.iter()
            .rev()
            .find(|r| r.tool_name == tool_name && r.success)
    }
}
```
