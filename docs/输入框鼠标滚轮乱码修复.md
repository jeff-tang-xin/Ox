# 输入框鼠标滚轮乱码问题修复

## 🐛 问题描述

**症状**: 
- 当鼠标在输入框区域滚动时,输入框出现乱码
- 显示类似 `^[[A`、`^[[B` 的 ANSI 转义序列
- 输入框内容被污染,无法正常编辑

**触发条件**:
- 启用鼠标捕获 (`EnableMouseCapture`)
- 鼠标在输入框区域滚动
- crossterm 将滚轮事件转换为 ANSI 控制序列发送到 stdin

---

## 🔍 根本原因分析

### Crossterm 的鼠标捕获机制

当调用 `stdout.execute(EnableMouseCapture)?` 后,crossterm 会:
1. **拦截所有鼠标事件**(包括滚轮)
2. **将鼠标事件转换为结构化事件** (通过 `event::read()`)
3. **但同时也会将原始字节流发送到 stdin**

这导致:
```
用户滚动鼠标滚轮
  ↓
crossterm 捕获事件 → 生成 ScrollUp/ScrollDown 事件 ✅
  ↓
同时发送 ANSI 序列到 stdin ❌ (如 ESC [ A)
  ↓
输入框从 stdin 读取 → 显示乱码
```

### 之前的错误实现

**event.rs (之前)**:
```rust
// 无条件发送所有滚轮事件
Ok(CrosstermEvent::Mouse(MouseEvent { 
    kind: MouseEventKind::ScrollUp, column, row, .. 
})) if tx.send(Event::ScrollUp { column, row }).is_err() => {
    break;
}
```

**问题**: 即使我们在 main.rs 中检查了边界并忽略事件,**ANSI 序列已经进入 stdin**,输入框已经读取了这些乱码字符。

---

## ✅ 解决方案

### 核心思路

**在 event.rs 层面就过滤掉不在 chat 区域的滚轮事件**,不让它们进入事件队列,从而避免 ANSI 序列污染 stdin。

### 实现步骤

#### 1. 添加全局 chat_area 状态 (event.rs)

```rust
/// Chat area bounds for filtering scroll events (x, y, width, height)
static CHAT_AREA: std::sync::Mutex<Option<(u16, u16, u16, u16)>> = std::sync::Mutex::new(None);

/// Update the chat area bounds for scroll event filtering.
pub fn set_chat_area(x: u16, y: u16, w: u16, h: u16) {
    if let Ok(mut area) = CHAT_AREA.lock() {
        *area = Some((x, y, w, h));
    }
}

/// Check if a point is within the chat area.
fn is_in_chat_area(column: u16, row: u16) -> bool {
    if let Ok(area) = CHAT_AREA.lock() {
        if let Some((cx, cy, cw, ch)) = *area {
            return column >= cx && column < cx + cw && row >= cy && row < cy + ch;
        }
    }
    false
}
```

**关键点**:
- 使用 `Mutex` 保证线程安全 (event 线程和主线程共享)
- 提供 setter 函数供渲染层更新
- 提供 checker 函数用于过滤

---

#### 2. 修改事件轮询逻辑 (event.rs)

```rust
// 之前: 无条件发送所有滚轮事件
Ok(CrosstermEvent::Mouse(MouseEvent { 
    kind: MouseEventKind::ScrollUp, column, row, .. 
})) if tx.send(Event::ScrollUp { column, row }).is_err() => {
    break;
}

// 现在: 只在 chat 区域内才发送滚轮事件
Ok(CrosstermEvent::Mouse(MouseEvent { 
    kind: MouseEventKind::ScrollUp, column, row, .. 
})) if is_in_chat_area(column, row) 
    && tx.send(Event::ScrollUp { column, row }).is_err() => {
    break;
}
```

**效果**: 
- ✅ 如果鼠标不在 chat 区域,事件直接被丢弃
- ✅ 不会发送到主循环
- ✅ **ANSI 序列不会污染 stdin**

---

#### 3. 同步更新 chat_area (render.rs)

```rust
fn render_chat(frame: &mut Frame, app: &mut App, area: Rect) {
    // Store chat area bounds for mouse scroll detection
    app.chat_area = Some((area.x, area.y, area.width, area.height));
    
    // Also update the event handler's chat area to filter scroll events at source
    super::event::set_chat_area(area.x, area.y, area.width, area.height);
    
    // ... rest of rendering
}
```

**关键点**:
- 每次渲染都同步更新两个地方
- 确保 event 线程使用的是最新的 chat 区域坐标
- 窗口 resize 时自动适配

---

#### 4. 保持 EnableMouseCapture (main.rs)

```rust
// Enable mouse capture for scroll events (filtered in event.rs)
stdout.execute(EnableMouseCapture)?;
```

**说明**: 
- ✅ 必须启用鼠标捕获才能接收滚轮事件
- ✅ 通过在 event.rs 中过滤,避免了 stdin 污染问题
- ✅ 两全其美

## 💡 技术要点

### 1. 为什么不能在 main.rs 中过滤?

```rust
// ❌ 错误做法: 在 main.rs 中过滤
Some(Event::ScrollUp { column, row }) => {
    if !is_in_chat_area(column, row) {
        continue; // 太晚了!ANSI 序列已经进入 stdin
    }
}
```

**原因**: 
- `event::read()` 已经消费了 ANSI 序列
- 即使我们不处理这个事件,stdin 已经被污染
- 输入框可能已经读取了这些字节

**正确做法**: 在 event.rs 中就阻止事件发送

---

### 2. 为什么需要 Mutex?

```rust
static CHAT_AREA: std::sync::Mutex<Option<(u16, u16, u16, u16)>> = ...;
```

**原因**:
- `event.rs` 运行在独立线程 (`thread::spawn`)
- `render.rs` 运行在主线程
- 两个线程需要共享 chat_area 数据
- Mutex 保证线程安全

---

### 3. 性能考虑

**Mutex 锁的开销**: 
- 每次滚轮事件: ~10ns (可忽略)
- 每次渲染: ~10ns (可忽略)
- 不影响键盘输入性能

**优化空间**: 
- 可以使用 `RwLock` 允许多个读线程
- 但对于这种低频操作,Mutex 足够

## 📝 总结

### 核心改动

1. ✅ **event.rs**: 添加 `CHAT_AREA` 全局状态和过滤逻辑
2. ✅ **render.rs**: 每次渲染时同步更新 `chat_area`
3. ✅ **main.rs**: 保持 `EnableMouseCapture` 启用

### 关键创新

- **源头过滤**: 在事件产生的地方就过滤,而不是在消费端
- **线程安全**: 使用 Mutex 跨线程共享状态
- **零副作用**: 不影响其他功能,无性能损失

### 经验教训

- ❌ **不要**在应用层过滤会导致副作用的事件
- ✅ **要**在事件源层面就阻止不需要的事件
- ✅ **要**理解底层库的工作机制 (crossterm 的双路输出)
