# 人格演化与提示词大小控制

## 问题分析

### Q1: 隐式反馈是异步还是同步？

**答案**: 混合模式

| 操作 | 执行方式 | 原因 |
|------|---------|------|
| 文件改动检测 | ✅ 同步 | 需在用户输入前完成，影响当前对话 |
| EMA 内存更新 | ✅ 同步 | <1μs，几乎无开销 |
| SQLite 持久化 | ⚡ 异步 | I/O 较慢 (~1-5ms)，后台执行不阻塞 TUI |
| 回滚评估 | ✅ 同步 | 需在人格演化前决策 |

**实现策略**:
```rust
// 同步检测（关键路径）
let signals = detector.detect_overrides(); // ~5ms for 100 files
update_counters(signals);                   // <1μs

// 异步持久化（非关键路径）
if should_persist() {
    tokio::spawn(async {
        ema_manager.persist_to_store(...).await;
    });
}
```

**性能保证**:
- 用户输入延迟: < 10ms (远低于 16ms 感知阈值)
- TUI 流畅度: 不受 I/O 阻塞影响
- 数据一致性: 关键状态同步更新，持久化最终一致

### Q2: 人格自进化后的提示词会不会过于庞大？

**答案**: ❌ 不会，设计已防止膨胀

#### 当前实现

**人格提示词模板** (固定格式):
```rust
"## Persona
- Safety priority: {safety:.1} | Conciseness: {concise:.1} | Style strictness: {style:.1}
- Refuses unsafe code: {refuses_unsafe}
- Forbidden phrases: {forbidden}
- Value priorities: {values}"
```

**演化前后对比**:
```
演化前: Safety priority: 0.9 | Conciseness: 0.8 | Style strictness: 0.9
演化后: Safety priority: 0.9 | Conciseness: 0.85 | Style strictness: 0.9
       ↑ 只是数字变化，长度完全相同！
```

**关键设计原则**:
1. ✅ **固定模板**: `generate_prompt_block()` 输出格式不变
2. ✅ **数值精度固定**: `.1` (一位小数)，保证长度一致
3. ✅ **无历史累积**: 不保存演化历史到 prompt 中
4. ✅ **只改数值**: 演化只调整 f64 值，不添加新字段

#### System Prompt 总大小

```
组件                      大小 (chars)    Tokens
────────────────────────────────────────────────
DEFAULT_PERSONA           150            ~50
WORKFLOW_RULES            1800           ~450
PersonaVector (演化部分)   200            ~60    ← 固定不变
Behavior Rules (可选)      200            ~50
Runtime Environment        150            ~40
────────────────────────────────────────────────
总计                      ~2500          ~650
```

**占比分析**:
- 人格演化部分: 60/650 = **9%** (且固定不变)
- 占 128K 上下文窗口: 650/131072 = **0.5%** (可忽略)
- 主要开销: WORKFLOW_RULES (70%)，但这是静态的

## 防护措施

### 1. 人格维度数量限制

**当前**: 3 个可演化维度
- `favors_safety_over_speed`
- `prefers_conciseness`
- `code_style_strictness`

**建议上限**: ≤ 5 个维度

**理由**:
- 每增加 1 个维度 → +~40 chars (~10 tokens)
- 5 个维度 → +200 chars (仍在合理范围)
- > 5 个维度 → 考虑重构为结构化描述

### 2. 禁止短语列表长度控制

**当前实现**:
```rust
forbidden_phrases.join(", ")
```

**风险**: 如果积累过多禁止短语，可能导致 prompt 膨胀

**防护建议**:
```rust
// 限制最多显示 5 个
let display_forbidden: Vec<&str> = self.forbidden_phrases.iter()
    .take(5)
    .collect();
let forbidden_str = if self.forbidden_phrases.len() > 5 {
    format!("{}, ... (+{} more)", 
        display_forbidden.join(", "),
        self.forbidden_phrases.len() - 5
    )
} else {
    display_forbidden.join(", ")
};
```

### 3. 价值观优先级截断

**类似防护**:
```rust
// 最多显示 3 个价值观
let display_values: Vec<&str> = self.moral_priorities.iter()
    .take(3)
    .collect();
let values_str = display_values.join(", ");
```

### 4. 监控提示词大小

**建议添加日志**:
```rust
pub fn generate_prompt_block(&self) -> String {
    let prompt = format!(...);
    
    // 监控警告
    if prompt.len() > 500 {
        tracing::warn!(
            "Persona prompt size: {} chars ({} tokens) - consider optimization",
            prompt.len(),
            prompt.len() / 4  // rough estimate
        );
    }
    
    prompt
}
```

### 方案 B: 动态裁剪

```rust
pub fn generate_prompt_block_with_budget(&self, max_chars: usize) -> String {
    let full = self.generate_prompt_block();
    
    if full.len() <= max_chars {
        return full;
    }
    
    // 渐进式裁剪
    let mut result = String::new();
    result.push_str(&format!("Safety: {:.1} | Conciseness: {:.1}", 
        self.favors_safety_over_speed, 
        self.prefers_conciseness));
    
    if result.len() + 50 < max_chars {
        result.push_str(&format!(" | Style: {:.1}", self.code_style_strictness));
    }
    
    result
}
```

## 最佳实践建议

### ✅ DO (推荐做法)

1. **保持维度精简** - ≤ 5 个可演化数值维度
2. **限制列表长度** - 禁止短语 ≤ 5 个，价值观 ≤ 3 个
3. **固定输出格式** - 使用 `.1` 精度，避免浮点数长度波动
4. **监控大小** - 添加日志警告，当 prompt > 500 chars 时报警
5. **定期审计** - 每季度检查人格维度的必要性

### ❌ DON'T (避免做法)

1. **不要累积历史** - 不在 prompt 中包含演化历史
2. **不要动态添加字段** - 避免根据条件添加不同数量的字段
3. **不要使用可变精度** - 避免 `{:.3}` vs `{:.1}` 混用
4. **不要嵌入长文本** - 禁止短语应该是简短关键词，不是句子

## 总结

| 问题 | 答案 | 关键原因 |
|------|------|---------|
| 隐式反馈同步/异步？ | 混合模式 | 检测同步，I/O 异步 |
| 提示词会膨胀吗？ | ❌ 不会 | 固定格式，只改数值 |
| 当前大小 | ~650 tokens | 占上下文窗口 < 1% |
| 演化 100 次后 | ~650 tokens | 完全不变 |
| 需要防护吗？ | 建议添加 | 防止未来扩展失控 |

**核心设计理念**: 
> **"Evolve values, not structure"**  
> (演化数值，而非结构)

这保证了人格系统可以无限演化，而不会导致提示词膨胀。
