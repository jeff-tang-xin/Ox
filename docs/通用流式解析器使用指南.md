# 通用流式响应解析器使用指南

## 📖 概述

Ox CLI 提供了一个**简洁高效的流式响应解析框架**,基于**两种核心协议**支持所有主流大模型:

### 两种核心协议

1. **OpenAI 协议** - 适用于:
   - ✅ OpenAI (GPT-4, GPT-3.5, o1)
   - ✅ DeepSeek (完全兼容 OpenAI 格式)
   - ✅ Groq, Together AI, SiliconFlow
   - ✅ 任何 OpenAI-API 兼容的提供商

2. **Anthropic 协议** - 适用于:
   - ✅ Anthropic Claude (Claude-3 Opus/Sonnet/Haiku)

> 💡 **设计哲学**: 市场上 95%+ 的 LLM 提供商都兼容这两种协议之一。DeepSeek、Groq 等虽然有自己的品牌,但 API 格式与 OpenAI 完全兼容,因此无需单独实现。

## 🏗️ 架构设计

```
┌─────────────────────────────────────┐
│  Layer 1: SSE Protocol Parser       │ ← sse::SseEventBuffer
│  - RFC 6455 标准兼容                 │
│  - 处理 data:, event:, 空行边界      │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Layer 2: JSON Stream Parser        │ ← sse::parse_json_values
│  - 解析完整/截断的 JSON              │
│  - 优雅处理网络中断                  │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Layer 3: Protocol Adapter          │ ← universal_adapter::*
│  - OpenAiAdapter (OpenAI 协议)       │
│  - AnthropicAdapter (Anthropic 协议) │
│  - 工厂函数自动选择                  │
└─────────────────────────────────────┘
```

## 🚀 快速开始

### 基本用法

```rust
use ox_core::llm::universal_adapter::{create_adapter, StreamAdapter};
use ox_core::llm::sse::{SseEventBuffer, parse_json_values};

// 1. 创建适配器 (根据提供商自动选择协议)
let mut adapter = create_adapter("openai");   // OpenAI 协议
// let mut adapter = create_adapter("anthropic"); // Anthropic 协议
// let mut adapter = create_adapter("deepseek");  // 也使用 OpenAI 协议

// 2. 初始化 SSE buffer
let mut sse_buffer = SseEventBuffer::new();

// 3. 处理流式数据
for line in stream_lines {
    if sse_buffer.push_line(line) {
        let data = sse_buffer.take_data();
        
        // 解析 JSON (可能包含多个值)
        for json_result in parse_json_values(&data) {
            if let Ok(json) = json_result {
                // 转换为通用事件
                let events = adapter.process_chunk(&json);
                
                for event in events {
                    match event {
                        LlmStreamEvent::TextDelta(text) => {
                            print!("{}", text);
                        }
                        LlmStreamEvent::ToolCallStart { id, name } => {
                            println!("Starting tool: {}", name);
                        }
                        LlmStreamEvent::Done { usage } => {
                            println!("Tokens used: {}", usage.total_tokens);
                        }
                        _ => {}
                    }
                }
            }
        }
    }
}
```

```rust
async fn stream_chat(...) -> Result<()> {
    let resp = client.post(url).send().await?;
    let mut stream = resp.bytes_stream();
    
    // 使用通用适配器
    let mut adapter = create_adapter(&self.provider_name);
    let mut sse_buffer = SseEventBuffer::new();
    
    while let Some(chunk) = stream.next().await {
        let chunk_str = String::from_utf8_lossy(&chunk?);
        
        for line in chunk_str.lines() {
            if sse_buffer.push_line(line) {
                let data = sse_buffer.take_data();
                
                for json_result in parse_json_values(&data) {
                    match json_result {
                        Ok(json) => {
                            let events = adapter.process_chunk(&json);
                            for event in events {
                                let _ = tx.send(event);
                            }
                        }
                        Err(e) => {
                            tracing::debug!("Skipping malformed JSON: {}", e);
                        }
                    }
                }
            }
        }
    }
    
    // Finalize
    for event in adapter.finalize() {
        let _ = tx.send(event);
    }
    
    Ok(())
}
```

## ✨ 核心优势

### 1. **统一的错误处理**
所有模型共享相同的截断 JSON 处理逻辑:
```rust
// parse_json_values 会自动跳过不完整的 JSON
// 不会因网络中断而 panic
```

### 2. **一致的 API**
无论使用哪个提供商,都返回相同的 `LlmStreamEvent`:
- `TextDelta(String)` - 文本片段
- `ToolCallStart { id, name }` - 工具调用开始
- `ToolCallArgumentsDelta { id, delta }` - 工具参数增量
- `ToolCallEnd { id }` - 工具调用结束
- `Done { usage }` - 完成 + Token 统计
- `Error(String)` - 错误信息

### 3. **易于扩展**

虽然目前只支持两种协议,但添加新协议非常简单:

## 📊 支持的协议

### OpenAI 协议格式
```json
{
  "choices": [{
    "index": 0,
    "delta": {"content": "Hello"},
    "finish_reason": null
  }],
  "usage": {"prompt_tokens": 10, "completion_tokens": 5, "total_tokens": 15}
}
```

**适用提供商**: OpenAI, DeepSeek, Groq, Together AI, SiliconFlow, 以及任何兼容 OpenAI API 的提供商。

### Anthropic 协议格式
```json
{
  "type": "content_block_delta",
  "index": 0,
  "delta": {
    "type": "text_delta",
    "text": "Hello"
  }
}
```

**适用提供商**: Anthropic Claude 系列。

### 工具调用 (OpenAI)
```json
{
  "choices": [{
    "delta": {
      "tool_calls": [{
        "index": 0,
        "id": "call_abc",
        "function": {
          "name": "file_write",
          "arguments": "{\"path\": \"test.txt\""
        }
      }]
    }
  }]
}
```

### 工具调用 (Anthropic)
```json
{
  "type": "content_block_start",
  "index": 0,
  "content_block": {
    "type": "tool_use",
    "id": "toolu_abc",
    "name": "file_write"
  }
}
```

## 📝 最佳实践

1. **始终使用 `parse_json_values`** 而非直接 `serde_json::from_str`
   - 自动处理数组、多值、截断等情况

2. **检查 `LlmStreamEvent::Error`**
   - 提供商特定的错误已转换为统一格式

3. **调用 `adapter.finalize()`**
   - 确保所有 pending 的工具调用都被正确关闭

4. **使用 `tracing::debug` 记录解析错误**
   - 不要忽略解析失败,但也不要让它中断流程

## 🔮 未来扩展

### 当前策略

目前只维护**两种核心协议适配器**:
- `OpenAiAdapter` - 覆盖 OpenAI 及所有兼容提供商
- `AnthropicAdapter` - 覆盖 Anthropic Claude

这个设计已经覆盖了市场上 95%+ 的 LLM 提供商。

### 何时添加新协议?

只有在以下情况才需要添加新的适配器:
1. 新提供商使用**完全不同**的 API 格式
2. 不兼容 OpenAI 或 Anthropic 协议
3. 有明确的市场需求

例如,如果 Google Gemini 推出后使用完全不同的格式,可以添加 `GeminiAdapter`。

### 如何判断是否需要新协议?

检查新提供商的 API 文档:
- ✅ 如果接口类似 `POST /v1/chat/completions` → 使用 `OpenAiAdapter`
- ✅ 如果 SSE 事件包含 `choices[].delta` → 使用 `OpenAiAdapter`
- ✅ 如果 SSE 事件包含 `type: "content_block_delta"` → 使用 `AnthropicAdapter`
- ❌ 如果完全不同于以上两种 → 考虑添加新适配器
