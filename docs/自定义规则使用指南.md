# 自定义规则使用指南

## 🎯 核心概念

**自定义规则**允许你覆盖 Ox 的默认行为，强制 LLM 遵守你的编码规范和项目约定。

---

## 📍 自定义规则在哪儿？

### 配置文件位置

```bash
~/.ox/config.toml
```

**Windows**: `C:\Users\<用户名>\.ox\config.toml`  
**Linux/Mac**: `~/.ox/config.toml`

---

## ⚙️ 怎么配置？

### 基础配置结构

```toml
[behavior_rules]
# ── 内置行为规则开关（可选）──
enforce_safe_code = true    # 安全代码检查
enforce_lint = true         # 运行 lint
enforce_format = true       # 格式化代码
enforce_tests = true        # 编写测试
enforce_all = true          # 启用所有内置规则

# ── 自定义规则（优先级更高）──
custom_rules = [
    "Rule 1",
    "Rule 2",
    "Rule 3"
]
```

## 📝 配置示例

### 示例 1: Rust 项目规范

```toml
[behavior_rules]
custom_rules = [
    "Always use Result<T, E> for error handling instead of unwrap() or expect()",
    "Prefer async/await over blocking operations for I/O",
    "Use serde derive macros for serialization/deserialization",
    "Follow Rust naming conventions: snake_case for functions/variables, PascalCase for types/enums",
    "Add doc comments (///) to all public functions and types",
    "Use thiserror crate for defining custom error types",
    "Prefer iterators and functional style over explicit loops when possible",
    "Avoid using .clone() unless necessary - prefer references (&T)"
]
```

### 示例 2: Python 项目规范

```toml
[behavior_rules]
custom_rules = [
    "Follow PEP 8 style guide for Python code",
    "Use type hints for all function parameters and return values",
    "Prefer list comprehensions over map/filter when appropriate",
    "Use f-strings for string formatting instead of .format() or %",
    "Add docstrings (Google style) to all functions and classes",
    "Use pathlib.Path instead of os.path for file operations",
    "Prefer dependency injection over global state",
    "Write unit tests for all new functions using pytest"
]
```

### 示例 3: Web API 项目规范

```toml
[behavior_rules]
custom_rules = [
    "All API endpoints must return JSON with consistent structure: {success, data, error}",
    "Use HTTP status codes correctly: 200 (OK), 201 (Created), 400 (Bad Request), 404 (Not Found), 500 (Server Error)",
    "Validate all input parameters before processing",
    "Log all errors with context information (request ID, user ID, timestamp)",

    "Never expose internal error messages to clients - use generic messages",

```

### 示例 4: 团队协作规范

```toml
[behavior_rules]
custom_rules = [
    "Write commit messages in conventional commits format: feat:, fix:, docs:, etc.",
    "Keep functions under 50 lines - split larger functions into smaller ones",
    "No commented-out code in production - delete it or move to TODO comments",
```

## 🔄 怎么执行？

### 执行流程

```
1. 用户启动 Ox
   ↓
2. 加载 ~/.ox/config.toml
   ↓
3. 读取 [behavior_rules] 配置
   ↓
4. 构建 System Prompt:
   - 如果有 custom_rules → 注入自定义规则
   - 如果无 custom_rules → 注入内置规则
   ↓
5. LLM 接收完整的 System Prompt
   ↓
6. LLM 生成代码时遵守这些规则
   ↓
7. 用户看到符合规范的代码
```

### 实际效果

**没有自定义规则**：
```rust
// LLM 可能生成
fn get_user(id: u32) -> User {
    let user = db.query(id).unwrap();  // ❌ 使用了 unwrap()
    user
}
```

**有自定义规则后**：
```toml
[behavior_rules]
custom_rules = [
    "Always use Result<T, E> for error handling instead of unwrap()"
]
```

```rust
// LLM 生成
use anyhow::Result;

fn get_user(id: u32) -> Result<User> {
    let user = db.query(id)?;  // ✅ 使用 ? 运算符
    Ok(user)
}
```

## ⚠️ 规则重复了怎么办？

### 优先级机制

Ox 有三层规则系统，优先级从低到高：

#### Layer 1: WORKFLOW_RULES（最低优先级，无法覆盖）

硬编码在 [system_prompt.rs](file:///F:/rust/Ox/crates/ox-core/src/context/system_prompt.rs#L80-L127) 中：

```
## Core Principles (MANDATORY)
1. Think Before Coding
2. Simplicity First
3. Surgical Changes
4. Goal-Driven Execution
```

**特点**：
- ✅ 始终存在，无法被自定义规则覆盖
- ✅ 提供基础的编码原则
- ⚠️ 可能与自定义规则冲突

---

#### Layer 2: Behavior Rules（中等优先级）

来自配置的 `[behavior_rules]`：

**情况 A：无自定义规则**
```toml
[behavior_rules]
enforce_all = true
```

生成的 prompt：
```
## Behavior Rules
- Never suggest code that bypasses safety checks
- Always run lint before declaring code complete
- Always format code before writing files
- Always write tests for new functions
```

**情况 B：有自定义规则**
```toml
[behavior_rules]
custom_rules = [
    "Always use Result<T, E> for error handling"
]
```

生成的 prompt：
```
## User-Defined Coding Rules (HIGHEST PRIORITY)

⚠️ These rules OVERRIDE any conflicting principles below.

These rules MUST be followed in ALL code you write:
1. Always use Result<T, E> for error handling
```

---

#### Layer 3: Memory Context（动态注入，与规则互补）

从记忆系统检索的知识（见 `/memory` 命令）：

```
Relevant context:
- 项目使用 PostgreSQL 14 数据库
- 偏好函数式编程风格
- 使用 thiserror 处理错误
```

**特点**：
- ✅ 提供项目特定知识
- ✅ 不与规则冲突，而是补充上下文
- ✅ 基于反馈持续优化

---

### 冲突处理策略

#### 场景 1: 自定义规则 vs WORKFLOW_RULES

**WORKFLOW_RULES 说**：
> "Simplicity First: Minimum code that solves the problem."

**自定义规则说**：
> "Always write comprehensive documentation for every function"

**结果**：
- LLM 会看到两个规则的冲突
- 由于自定义规则标题是 **"HIGHEST PRIORITY"**，LLM 倾向于遵循自定义规则
- 但可能会产生困惑

**解决方案**：
1. **避免直接冲突** - 不要定义与核心原则矛盾的规则
2. **明确意图** - 在规则中说明为什么需要这样做

```toml
# ❌ 不好的规则（与 Simplicity First 冲突）
custom_rules = [
    "Always write comprehensive documentation"
]

# ✅ 好的规则（说明原因）
custom_rules = [
    "Write doc comments for public APIs to improve team collaboration"
]
```

---

#### 场景 2: 多个自定义规则之间冲突

**示例**：
```toml
custom_rules = [
    "Keep functions under 20 lines",
    "Write comprehensive error handling with detailed logging"
]
```

**问题**：详细的错误处理可能导致函数超过 20 行。

**解决方案**：
```toml
custom_rules = [
    "Keep functions under 50 lines (split complex logic into helper functions)",
    "Write clear error handling, but extract complex validation into separate functions"
]
```

## 💡 最佳实践

### ✅ DO（推荐做法）

1. **规则要具体、可执行**
   ```toml
   # ✅ 好
   custom_rules = [
       "Use Result<T, E> instead of unwrap()",
       "Add type hints to all function parameters"
   ]
   
   # ❌ 不好（太模糊）
   custom_rules = [
       "Write good code",
       "Follow best practices"
   ]
   ```

2. **规则数量控制在 5-10 条**
   - 太多规则会让 LLM 困惑
   - 优先最重要的规范

3. **定期审查和更新规则**
   ```bash
   # 查看当前学到的知识
   /memory
   
   # 如果发现 Ox 违反了某个规则
   /feedback bad  # （注意：当前 /feedback bad 已移除，需要手动调整规则）
   
   # 更新 config.toml 中的规则
   ```

4. **规则要与项目技术栈匹配**
   ```toml
   # Rust 项目
   custom_rules = [
       "Use cargo fmt before committing",
       "Run cargo clippy --all-targets"
   ]
   
   # Python 项目
   custom_rules = [
       "Run black for formatting",
       "Run flake8 for linting"
   ]
   ```

---

### ❌ DON'T（避免做法）

1. **不要定义与核心原则直接冲突的规则**
   ```toml
   # ❌ 与 "Think Before Coding" 冲突
   custom_rules = [
       "Always implement features immediately without asking questions"
   ]
   ```

2. **不要定义无法验证的规则**
   ```toml
   # ❌ 无法验证
   custom_rules = [
       "Write elegant code"
   ]
   
   # ✅ 可验证
   custom_rules = [
       "Keep cyclomatic complexity under 10"
   ]
   ```

3. **不要过度约束**
   ```toml
   # ❌ 太多细节，限制 LLM 灵活性
   custom_rules = [
       "Use 2 spaces for indentation",
       "Put opening brace on same line",
       "Add blank line after imports",
       "Use single quotes for strings",
       # ... 20 more rules
   ]
   
   # ✅ 关注重要规范
   custom_rules = [
       "Follow standard Rust formatting (cargo fmt)",
       "Use Result for error handling",
       "Add tests for new functionality"
   ]
   ```

## 🔍 调试技巧

### 1. 查看当前生效的规则

运行 Ox 后，你可以问它：

```
User: "What coding rules are you following?"

Assistant: "I'm following these rules:
1. Use Result<T, E> for error handling
2. Prefer async/await over blocking operations
3. ..."
```

### 2. 测试规则是否生效

```toml
# config.toml
[behavior_rules]
custom_rules = [
    "Always add a comment explaining the purpose of each function"
]
```

```
User: "Write a function to calculate factorial"

# 期望输出
Assistant: "
// Calculate factorial of n using recursion
fn factorial(n: u64) -> u64 {
    if n <= 1 {
        1
    } else {
        n * factorial(n - 1)
    }
}
"
```

### 3. 检查规则冲突

如果你发现 LLM 的行为不符合预期：

1. **检查 config.toml 语法**
   ```bash
   # 验证 TOML 格式
   cat ~/.ox/config.toml | python -m toml
   ```

2. **简化规则**
   - 暂时只保留 1-2 条规则
   - 逐步添加，观察效果

3. **查看日志**
   ```bash
   # 启用调试模式
   ox --debug
   ```

## 📊 完整配置示例

```toml
# ~/.ox/config.toml

[general]
lang = "zh"

[models]
default = "gpt-4o"

[models.providers.openai]
api_key = "sk-..."

[context]
max_history_turns = 20

# ── 自定义规则配置 ──
[behavior_rules]
# 禁用内置规则（因为我们有自定义规则）
enforce_all = false

# 定义项目特定的编码规范
custom_rules = [
    "Use Result<T, anyhow::Error> for all fallible operations",
    "Prefer async/await for I/O operations (database, HTTP, file system)",
    "Add #[derive(Debug)] to all custom types for better error messages",
    "Use thiserror crate to define domain-specific error types",
    "Write integration tests for all public API endpoints",
    "Log errors with context: use tracing::error! with request_id and user_id",
    "Validate all user input before processing (use validator crate)",
    "Use dependency injection for database connections and external services",
    "Document public APIs with /// doc comments including examples",
    "Run cargo fmt and cargo clippy before committing changes"
]

[memory]
max_nodes = 1000
```

## 🎉 总结

### 核心价值

1. **个性化** - 适应你的编码风格和项目需求
2. **一致性** - 确保 Ox 始终遵循团队规范
3. **高效** - 减少代码审查时的返工

### 关键要点

- ✅ 自定义规则存储在 `~/.ox/config.toml`
- ✅ 有自定义规则时，完全覆盖内置规则
- ✅ 自定义规则优先级高于 WORKFLOW_RULES（通过标题强调）
- ✅ 规则要具体、可验证、数量适中（5-10 条）
- ✅ 定期审查和更新规则

### 快速开始

```bash
# 1. 编辑配置文件
nano ~/.ox/config.toml

# 2. 添加自定义规则
[behavior_rules]
custom_rules = [
    "Your rule 1",
    "Your rule 2"
]

# 3. 重启 Ox
ox

# 4. 测试规则是否生效
User: "帮我写一个函数..."
```
