# 流式响应协议设计决策

## 📋 决策摘要

**决策**: Ox CLI 只维护**两种核心流式响应协议适配器**

- ✅ **OpenAI 协议** - `OpenAiAdapter`
- ✅ **Anthropic 协议** - `AnthropicAdapter`

**理由**: 
1. 市场覆盖率 >95%
2. 代码简洁易维护
3. DeepSeek/Groq 等兼容 OpenAI 格式

---

## 🎯 设计原则

### 1. 协议优先,而非品牌优先

我们按**协议格式**分类,而非按**提供商品牌**分类:

```
❌ 错误思维: "需要为 DeepSeek 创建单独的适配器"
✅ 正确思维: "DeepSeek 使用 OpenAI 协议,复用 OpenAiAdapter"
```

### 2. 80/20 法则

- **20% 的代码** (2个适配器) 覆盖 **80%+ 的场景**
- 剩余的特殊情况按需扩展

### 3. 避免过度工程化

不预先为未出现的协议创建适配器框架。

## 🔍 如何判断新提供商使用哪个协议?

### 检查清单

1. **API 端点**
   - ✅ `POST /v1/chat/completions` → OpenAI 协议
   - ✅ `POST /v1/messages` (streaming) → Anthropic 协议

2. **SSE 事件格式**
   - ✅ 包含 `choices[].delta.content` → OpenAI 协议
   - ✅ 包含 `type: "content_block_delta"` → Anthropic 协议

3. **工具调用格式**
   - ✅ `tool_calls[].function.arguments` → OpenAI 协议
   - ✅ `content_block.type: "tool_use"` → Anthropic 协议

### 快速测试

```bash
# 发送一个简单的流式请求,观察 SSE 数据格式
curl -N https://api.example.com/v1/chat/completions \
  -H "Authorization: Bearer YOUR_KEY" \
  -d '{"model":"test","messages":[{"role":"user","content":"hi"}],"stream":true}'
```

如果输出类似:
```
data: {"choices":[{"delta":{"content":"Hello"}}]}
```
→ 使用 `OpenAiAdapter`

如果输出类似:
```
event: content_block_delta
data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"Hello"}}
```
→ 使用 `AnthropicAdapter`

## 💡 实际案例

### 案例 1: DeepSeek

**问题**: "DeepSeek 是中国的大模型,是否需要单独适配器?"

**分析**:
```json
// DeepSeek API 响应
{
  "choices": [{
    "delta": {"content": "你好"}
  }]
}
```

**结论**: ✅ 使用 `OpenAiAdapter`,无需额外工作

### 案例 2: Groq

**问题**: "Groq 速度超快,格式是否不同?"

**分析**:
```json
// Groq API 响应
{
  "choices": [{
    "delta": {"content": "Fast response!"}
  }]
}
```

**结论**: ✅ 使用 `OpenAiAdapter`,只需配置不同的 `base_url`

### 案例 3: 假设的 Gemini (未来)

**问题**: "如果 Gemini 使用完全不同的格式怎么办?"

**分析**:
```json
// 假设的 Gemini 响应
{
  "candidates": [{
    "content": {
      "parts": [{"text": "Hello"}]
    }
  }]
}
```

**结论**: ❌ 需要创建 `GeminiAdapter`,因为格式完全不同

## 🛠️ 实施指南

### 添加新提供商 (使用现有协议)

只需在 `config.toml` 中配置:

```toml
[models.providers.deepseek]
api_key = "sk-..."
base_url = "https://api.deepseek.com/v1"
# 自动使用 OpenAiAdapter,无需修改代码
```

### 添加新协议适配器 (罕见情况)

1. 实现 `StreamAdapter` trait
2. 在 `create_adapter()` 中注册
3. 添加测试用例
4. 更新本文档

参考: [universal_adapter.rs](file:///F:/rust/Ox/crates/ox-core/src/llm/universal_adapter.rs)

## 📈 效益分析

### 代码简化

- **之前**: 可能为每个提供商创建独立适配器 (5-10个)
- **现在**: 只维护 2 个核心适配器
- **减少**: ~80% 的解析代码

### 维护成本

- **之前**: 每次新提供商都需要适配、测试、文档
- **现在**: 95% 的情况只需配置 `base_url`

### 用户体验

- **之前**: 需要查阅每个提供商的兼容性说明
- **现在**: 自动识别,开箱即用

---

## ⚠️ 注意事项

### 边界情况

1. **部分兼容**: 某些提供商可能 90% 兼容,但有细微差异
   - 解决方案: 在 `OpenAiAdapter` 中添加条件逻辑

2. **版本演进**: 提供商可能改变 API 格式
   - 解决方案: 监控 breaking changes,必要时 fork 适配器

3. **性能差异**: 不同提供商的流式速度不同
   - 解决方案: 这是网络层问题,与协议无关

### 何时重新评估?

- 当市场上出现**大量**使用新协议的提供商时
- 当某个新协议的市场份额 >10% 时
- 当用户频繁请求支持某个新协议时
