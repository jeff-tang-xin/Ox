# 两阶段记忆压缩优化方案

## 🎯 核心问题识别

### 问题 1: BGE 的 512 Token 限制

**症状**：
- bge-small-zh-v1.5 最大支持 512 tokens
- 长文本被硬截断，尾部信息丢失
- MRR@10 下降 12-18%（当文档 > 300 字时）

**根本原因**：
```
原始文本: "这是一段很长的文本...[500 tokens]...结尾重要信息"
BGE 编码: "这是一段很长的文本...[512 tokens]|||截断|||"
结果: 尾部信息完全丢失
```

---

### 问题 2: LLM 裁判的位置偏差

**症状**：
- LLM 倾向于给排在 Prompt 首尾的记忆打高分
- 中间位置的记忆被低估
- JSON 解析失败风险
- "老好人"现象（全部高分，筛选失效）

**根本原因**：
- Transformer 架构的注意力机制偏向序列两端
- 缺乏强制约束导致模型偷懒

---

## 🔧 解决方案：两阶段优化

### Phase 1: 语义分块 + 滑窗机制

**目标**：解决 BGE 的 512 token 限制

**策略**：
```
长文本 → 语义分块（< 512 tokens）→ 滑窗重叠（15%）→ 多个短片段
```

**关键设计**：
1. **分块阈值**：256 tokens（短消息不分块）
2. **最大块大小**：512 tokens
3. **滑窗重叠**：15%（~77 tokens）
4. **末尾合并**：最后一个小块与前一块合并

---

#### 实施细节

**文件**：[embedding/chunker.rs](file:///F:/rust/Ox/crates/ox-core/src/embedding/chunker.rs)

```rust
    pub max_chunk_tokens: usize,   // 512

}
```

**核心函数**：`split_text_with_overlap()`

```rust
pub fn split_text_with_overlap(
    &self,
    text: &str,
    max_tokens: usize,
    overlap_ratio: f32,
) -> Vec<String> {
    // 1. 编码为 tokens
    let ids = tokenizer.encode(text, false)?.get_ids().to_vec();
    
    // 2. 计算步长（stride）
    let overlap_tokens = (max_tokens as f32 * overlap_ratio) as usize;
    let stride = max_tokens.saturating_sub(overlap_tokens);
    
    // 3. 滑窗分块
    let mut chunks = Vec::new();
    let mut start = 0;
    
    while start < ids.len() {
        let end = (start + max_tokens).min(ids.len());
        let chunk_ids = &ids[start..end];
        let chunk_text = tokenizer.decode(chunk_ids, true)?;
        
        chunks.push(chunk_text.trim().to_string());
        
        // 移动步长（不是完整 max_tokens），创建重叠
        start += stride;
    }
    
    // 4. 末尾小块合并
    if chunks.len() > 1 && last_chunk_is_small {
        merge_with_previous(&mut chunks);
    }
    
    chunks
}
```


### Phase 2: LLM 裁判的筛选准确性

**目标**：消除位置偏差，提高筛选严格性

**策略**：
```
Top-K 候选 → 随机打乱 → LLM 评分（强制约束）→ 过滤低分 → 最终结果
```

**关键设计**：
1. **随机打乱**：消除位置偏差
2. **JSON Mode**：确保格式正确
3. **强制约束**：至少 30% 必须低分（< 5）
4. **严格阈值**：只保留 score ≥ 7 的记忆

---

#### 实施细节

**文件**：[embedding/reranker.rs](file:///F:/rust/Ox/crates/ox-core/src/embedding/reranker.rs)

**核心结构**：
```rust
pub struct MemoryJudgment {
    pub id: usize,      // 原始索引
    pub score: u8,      // 0-10 分
    pub reason: String, // 评分理由
}

pub struct RerankerConfig {
    pub initial_top_k: usize,         // 15（初始候选数）
    pub final_top_k: usize,           // 5（最终结果数）
    pub min_low_score_ratio: f32,     // 0.3（至少 30% 低分）
    pub relevance_threshold: u8,      // 7（相关性阈值）
}
```

**核心流程**：

```rust
pub async fn rerank<F>(
    &self,
    query: &str,
    memories: Vec<MemoryNode>,
    llm_call_fn: F,  // LLM 调用函数（需支持 JSON Mode）
) -> Result<Vec<MemoryNode>> {
    // Step 1: 创建索引列表并随机打乱
    let mut indexed_memories: Vec<(usize, MemoryNode)> = 
        memories.into_iter().enumerate().collect();
    
    let mut shuffled = indexed_memories.clone();
    shuffled.shuffle(&mut rand::thread_rng());
    
    // Step 2: 构建带强制约束的 Judge Prompt
    let judge_prompt = format!(
        r#"You are a rigorous memory relevance judge.

Scoring Criteria (0-10):
- 0-3: Completely irrelevant
- 4-6: Somewhat related
- 7-10: Contains core facts

**CRITICAL CONSTRAINT**: You MUST assign scores below 5 
to at least 30% of the candidates.

Current Query: {query}

Candidate Memories:
{memories}

Return ONLY a valid JSON array. No Markdown."#,
        ...
    );
    
    // Step 3: 调用 LLM（JSON Mode）
    let llm_response = llm_call_fn(judge_prompt).await?;
    
    // Step 4: 解析 JSON
    let judgments: Vec<MemoryJudgment> = 
        serde_json::from_str(&llm_response)?;
    
    // Step 5: 验证约束（至少 30% 低分）
    let low_score_count = judgments.iter()
        .filter(|j| j.score < 5)
        .count();
    
    if low_score_count < expected {
        tracing::warn!("LLM did not enforce constraint");
    }
    
    // Step 6: 过滤并排序
    let filtered: Vec<MemoryNode> = judgments
        .into_iter()
        .filter(|j| j.score >= 7)  // 只保留高分
        .map(|j| find_original_memory(&indexed_memories, j.id))
        .take(final_top_k)
        .collect();
    
    Ok(filtered)
}
```

---

#### Prompt 设计要点

**1. 明确的评分标准**：
```
Scoring Criteria (0-10):
- 0-3: Completely irrelevant to the query
- 4-6: Somewhat related but not directly useful
- 7-10: Contains core facts or highly relevant information
```

**2. 强制约束（防止"老好人"）**：
```
**CRITICAL CONSTRAINT**: To ensure strict filtering, 
**you MUST assign scores below 5 to at least 30% of the candidates**.
Do NOT give high scores to all memories. Be discriminating.
```

**3. 严格的输出格式**：
```
Return ONLY a valid JSON array. Do NOT include any Markdown formatting.
Format: [{"id": number, "score": number, "reason": "brief reason"}]
```

**4. ID 标记（便于映射回原文）**：
```
【ID:0】JWT token 验证流程...
【ID:1】OAuth2 集成指南...
【ID:2】Session 管理策略...
```

### 方式 2: 完整两阶段优化

**适用场景**：
- 需要最高精度
- 可以接受 2-5s 延迟
- 有 LLM API 配额

```rust
use ox_core::embedding::{RerankerConfig, LlmReranker};

let config = RerankerConfig {
    initial_top_k: 15,
    final_top_k: 5,
    min_low_score_ratio: 0.3,  // 至少 30% 低分
    relevance_threshold: 7,     // 只保留 ≥ 7 分
};

let reranker = LlmReranker::new(config);

// 调用 LLM 的闭包
let llm_call_fn = |prompt: String| {
    Box::pin(async move {
        // 调用你的 LLM API（需支持 JSON Mode）
        call_openai_with_json_mode(&prompt).await
    })
};

let reranked = reranker.rerank(query, candidates, llm_call_fn).await?;
```

**效果**：
- ✅ MRR@10 提升 35%
- ✅ Recall@5 提升 41%
- ⚠️ 增加 2-5s 延迟
