# 多模式智能体系统架构设计

## 📋 概述

本系统实现了**插件化的多模式智能体框架**，支持三种工作模式的无缝切换和扩展：

1. **Free Mode（自由探索模式）** - 默认的开放式对话和编程辅助
2. **Spec Mode（规范模式）** - 结构化的任务规划和工作流
3. **Council Mode（会议模式）** - 多智能体辩论和决策

## 🏗️ 架构设计

### 分层架构

```
┌─────────────────────────────────────────────────────┐
│           Agent Runtime (运行时)                      │
├─────────────────────────────────────────────────────┤
│  Registry System (注册中心)                           │
│  ├─ ModeRegistry      → 模式注册                      │
│  ├─ WorkflowRegistry  → 工作流注册                    │
│  ├─ ToolRegistry      → 工具注册                      │
│  ├─ ModelRegistry     → 模型注册                      │
│  ├─ StateRegistry     → 状态注册 (Session/Workflow)   │
│  └─ InterventionRegistry → 人工介入点注册             │
├─────────────────────────────────────────────────────┤
│  Engine Core (引擎核心)                               │
│  ├─ WorkflowEngine    → 强制执行步骤 + 验证           │
│  ├─ SessionManager    → Session 生命周期管理          │
│  ├─ InterventionManager → 人工介入处理                │
│  ├─ ContextBuilder    → 上下文构建                    │
│  └─ MessageRouter     → 消息路由                      │
├─────────────────────────────────────────────────────┤
│  Infrastructure (基础设施 - 共享)                     │
│  ├─ LLM Client / Memory / Compression / Cost        │
└─────────────────────────────────────────────────────┘
```

### 核心设计理念

1. **Mode（模式）** = Workflow + Rules + UI Behavior
2. **Workflow（工作流）** = 有序的步骤集合 + 转换规则
3. **Tool（工具）** = 可插拔的能力模块
4. **Model（模型）** = 可切换的 LLM 后端
5. **State（状态）** = 可持久化的运行状态

## 🔧 核心组件

### 1. Workflow Engine (`agent/engine.rs`)

**职责**：强制执行分步工作流，验证每步完成状态

```rust
pub struct WorkflowEngine {
    workflows: HashMap<String, Workflow>,       // 已注册的工作流
    current_workflow: Option<Workflow>,         // 当前激活的工作流
    session_state: Arc<Mutex<SessionState>>,    // Session 状态追踪
    intervention_manager: Arc<Mutex<InterventionManager>>, // 人工介入管理
}
```

**关键方法**：
- `activate_workflow(workflow_id)` - 激活指定工作流
- `current_step()` - 获取当前执行步骤
- `requires_user_confirmation()` - 检查是否需要用户确认
- `allows_tool_execution()` - 检查是否允许工具调用
- `validate_current_step()` - 验证当前步骤是否完成
- `advance_step()` - 推进到下一步

### 2. Session State (`agent/session.rs`)

**职责**：管理每个会话的状态，支持 Session 切换和持久化

```rust
pub struct SessionState {
    pub session_id: String,
    pub current_mode: String,              // 当前模式 (free/spec/council)
    pub current_workflow: String,          // 当前工作流 ID
    pub current_step_index: usize,         // 当前步骤索引
    pub awaiting_user_confirmation: bool,  // 是否等待用户确认
    pub variables: HashMap<String, String>, // Session 变量存储
    pub message_count: usize,              // 消息计数（用于压缩）
    pub is_active: bool,                   // 是否为活跃 Session
}
```

**关键特性**：
- 支持多 Session 切换
- Session 状态持久化
- 变量存储（用于工作流验证）

### 3. Intervention Manager (`agent/intervention.rs`)

**职责**：管理工作流中的人工介入点

```rust
pub enum InterventionType {
    Confirmation { message, allow_reject },  // 用户确认
    InputRequired { prompt, default_value }, // 用户输入
    ContentEdit { content_type, content },   // 内容编辑
    SkipOption { step_name, warning },       // 跳过选项
}
```

**工作流程**：
```
工作流到达需要确认的步骤
  ↓
InterventionManager 创建介入请求
  ↓
UI 显示给用户（确认/拒绝/输入/跳过）
  ↓
用户响应
  ↓
InterventionManager 处理响应并返回行动指令
  ↓
WorkflowEngine 根据行动继续或中止
```

### 4. Workflow Definition (`agent/workflow.rs`)

**定义工作流的步骤序列**：

```rust
pub struct WorkflowStep {
    pub id: String,
    pub name: String,
    pub description: String,
    pub requires_user_confirmation: bool,  // 是否需要用户确认
    pub allow_tool_execution: bool,        // 是否允许工具调用
    pub step_prompt: String,               // 步骤提示词（注入系统提示）
    pub validator_name: Option<String>,    // 验证器名称
}
```

## 📊 三种模式的工作流

### Spec Mode 工作流（6步强制流程）

```
Step 1: Requirement Analysis
  ↓ 自动
Step 2: Generate spec.md
  ↓ 代码验证文件存在
Step 3: Await Spec Confirmation ⚠️ USER CONFIRMATION REQUIRED
  ↓ 用户确认后
Step 4: Generate task.md
  ↓ 代码验证文件存在
Step 5: Await Task Confirmation ⚠️ USER CONFIRMATION REQUIRED
  ↓ 用户确认后
Step 6: Execute Code ✅ TOOLS ALLOWED
```

**关键约束**：
- Step 3 和 Step 5 **必须**等待用户确认
- Step 2 和 Step 4 通过验证器检查文件是否存在
- Step 6 之前**禁止**工具调用

### Council Mode 工作流（6步辩论流程）

```
Step 1: Topic Definition ⚠️ USER CONFIRMATION
  ↓
Step 2: Proposal Phase ❌ NO TOOLS
  ↓
Step 3: Review Phase ❌ NO TOOLS
  ↓
Step 4: Rebuttal Phase ❌ NO TOOLS
  ↓
Step 5: Arbitration ❌ NO TOOLS
  ↓
Step 6: Conclusion ⚠️ USER CONFIRMATION
```

**关键约束**：
- 全程禁用工具调用（内部对话）
- Step 1 和 Step 6 需要用户确认

### Free Mode 工作流（单步无限制）

```
Step 1: Free Interaction ✅ FULL ACCESS
```

## 🔄 Session 状态管理

### Session 切换流程

```
用户执行 /new 命令
  ↓
保存当前 Session 状态到 StateRegistry
  ↓
创建新的 SessionState
  ↓
重置工作流到初始状态
  ↓
激活新 Session
```

### Session 状态持久化

每个 Session 包含：
- 当前模式和步骤
- 工作流变量（如 `spec_confirmed`, `task_type`）
- 消息计数（用于压缩追踪）

## 🎯 人工介入机制

### 介入点类型

1. **Confirmation** - 用户确认/拒绝
   ```
   "Spec 文件已生成，是否确认并继续？"
   [确认] [拒绝]
   ```

2. **InputRequired** - 用户输入
   ```
   "请输入讨论主题："
   [________________]
   ```

3. **ContentEdit** - 内容编辑
   ```
   "请审查并编辑 spec.md 内容："
   [编辑器界面]
   ```

4. **SkipOption** - 跳过步骤
   ```
   "是否跳过 Spec 生成？"
   [跳过] [继续]
   ```

### 介入响应处理

```rust
pub enum InterventionAction {
    Proceed,                 // 继续下一步
    Abort,                   // 中止当前操作
    ContinueWithInput(String), // 使用用户输入继续
    SkipStep,                // 跳过当前步骤
    CancelWorkflow,          // 取消整个工作流
}
```

## 🔌 扩展性设计

### 添加新模式

1. 定义新的 Workflow（在 `workflow.rs` 中）
2. 注册到 WorkflowEngine
3. （可选）添加自定义验证器到 StateRegistry

```rust
// 示例：添加 Code Review 模式
let mut review_workflow = Workflow::new("review_workflow", "Code Review");
review_workflow.add_step(WorkflowStep::new("analyze", "Analyze Code", "..."));
review_workflow.add_step(WorkflowStep::new("suggest", "Suggest Improvements", "..."));
engine.register_workflow(review_workflow);
```

### 添加新工具

工具已经在 `ToolRegistry` 中注册，新模式可以通过 `allow_tool_execution` 字段控制是否可用。

### 添加新模型

模型在 `ModelRegistry` 中注册，可以随时切换。

## 🎓 使用示例

### 激活 Spec 模式

```
用户: /spec 实现用户认证功能

系统: 
  1. 激活 spec_workflow
  2. 设置 Session 状态: mode=spec, step=0
  3. LLM 收到增强的系统提示词:
     - TASK_TYPE_PROMPT
     - Step 1 的具体指令
  4. LLM 分析任务类型并分类
```

### 用户确认流程

```
LLM: 已生成 spec.md，请确认是否继续？

UI 显示:
  ┌─────────────────────────────────┐
  │ Spec Confirmation Required      │
  │                                 │
  │ 已生成规范文件 .ox/spec.md      │
  │ 是否确认并继续生成任务计划？     │
  │                                 │
  │  [✅ 确认]  [❌ 拒绝]  [✏️ 编辑] │
  └─────────────────────────────────┘

用户点击: ✅ 确认

系统:
  1. InterventionManager 接收 Confirmed 响应
  2. 返回 InterventionAction::Proceed
  3. WorkflowEngine 推进到 Step 4
  4. 设置 session.variables["spec_confirmed"] = "true"
  5. LLM 继续生成 task.md
```

## 🔍 关键技术点

1. **状态机驱动** - 所有模式转换通过状态机管理
2. **代码级强制** - 不是依赖 LLM 遵守规则，而是代码强制执行
3. **可插拔设计** - 新模式只需注册 Workflow 即可
4. **Session 隔离** - 每个 Session 独立状态，切换不丢失
5. **人工介入优先** - 关键步骤必须用户确认
