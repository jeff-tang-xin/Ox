# 文件写入乱码防护 - 实施完成

## 🎯 问题描述

**用户反馈**: "大模型改文件改出乱码了"

**根本原因**: 
- `file_write` 和 `file_patch` 工具没有任何内容验证
- LLM 可能生成编码错误的内容
- 可能包含不可打印字符、乱码、二进制数据
- 一旦写入文件，会导致文件损坏

## ✅ 解决方案

在文件写入前添加**5 层内容验证**，阻止乱码和损坏的内容被写入文件。

## 🛡️ 防护机制

### Check 1: UTF-8 编码验证

**检测**: 验证内容是否为有效的 UTF-8 编码

```rust
if !content.is_ascii() && !String::from_utf8(content.as_bytes().to_vec()).is_ok() {
    return Err("❌ Invalid Content: File content contains invalid UTF-8 encoding\n\n\
                💡 Possible causes:\n\
                • Content was copied from a corrupted source\n\
                • Binary data was included by mistake\n\
                • Encoding mismatch (e.g., GBK vs UTF-8)\n\n\
                📝 Please verify the content encoding and retry.".to_string());
}
```

**防护场景**:
- LLM 生成的内容包含无效 UTF-8 序列
- GBK/GB2312 编码的内容混入
- 二进制数据被当作文本处理

---

### Check 2: 不可打印字符比例检测

**检测**: 统计不可打印字符的比例（阈值：2%）

```rust
let non_printable_count = content
    .chars()
    .filter(|c| {
        !c.is_whitespace()
            && !c.is_ascii_graphic()
            && !c.is_ascii_punctuation()
            && !matches!(*c, '\n' | '\r' | '\t')
    })
    .count();

let non_printable_ratio = non_printable_count as f64 / total_chars as f64;

if non_printable_ratio > 0.02 {
    return Err(format!(
        "❌ Suspicious Content: {:.1}% of content contains non-printable characters ({}/{} chars)\n\n\
         💡 This usually indicates:\n\
         • Garbled/corrupted text (乱码)\n\
         • Binary data accidentally included\n\
         • Encoding conversion errors\n\n\
          Please verify the content is valid text and retry.",
        non_printable_ratio * 100.0,
        non_printable_count,
        total_chars
    ));
}
```

**防护场景**:
- 大量不可见字符（如 `\x00`, `\x01`, `\x1F`）
- 二进制数据混入文本
- 乱码通常包含高比例的不可打印字符

**阈值设计**:
- 2% 是一个合理的阈值
- 允许一些源代码包含特殊字符
- 超过 2% 通常意味着内容有问题

---

### Check 3: 乱码模式检测

**检测**: 识别常见的乱码模式

```rust
fn contains_garbled_patterns(content: &str) -> bool {
    // Pattern 1: Consecutive non-printable characters (≥5)
    let mut consecutive_non_printable = 0;
    for c in content.chars() {
        if !c.is_whitespace() && !c.is_ascii_graphic() && !c.is_ascii_punctuation() {
            consecutive_non_printable += 1;
            if consecutive_non_printable >= 5 {
                return true;
            }
        } else {
            consecutive_non_printable = 0;
        }
    }
    
    // Pattern 2: Mixed CJK and garbage characters
    let mut mixed_sequence = 0;
    for c in content.chars() {
        if (c.is_ascii() && !c.is_alphanumeric() && !c.is_whitespace())
            || (c.is_ascii_control() && !matches!(c, '\n' | '\r' | '\t'))
        {
            mixed_sequence += 1;
            if mixed_sequence >= 10 {
                return true;
            }
        } else {
            mixed_sequence = 0;
        }
    }
    
    false
}
```

**防护场景**:
- 连续 5 个以上不可打印字符
- 连续 10 个以上特殊控制字符
- 典型的编码错误模式

---

### Check 4: Null 字节检测

**检测**: 检查是否包含 `\x00`（空字节）

```rust
if content.contains('\x00') {
    return Err("❌ Corrupted Content: File contains null bytes (\\x00)\n\n\
                💡 This indicates:\n\
                • File corruption\n\
                • Binary data mixed with text\n\
                • Encoding errors\n\n\
                 Please verify and regenerate the content.".to_string());
}
```

**防护场景**:
- 文件损坏的典型标志
- 二进制数据被当作文本
- C 字符串的终止符泄漏

---

### Check 5: 替换字符检测

**检测**: 检查是否包含 U+FFFD（ 替换字符）

```rust
if content.contains('\u{FFFD}') {
    return Err("❌ Encoding Errors Detected: Content contains replacement characters (U+FFFD)\n\n\
                💡 This means:\n\
                • Original text had invalid encoding\n\
                • Conversion between encodings failed\n\
                • Data was corrupted during transfer\n\n\
                 Please use the original source with correct encoding.".to_string());
}
```

**防护场景**:
- 编码转换失败（如 GBK → UTF-8 失败）
- 无效的字节序列被替换为 ``
- 数据传输过程中的损坏

### 验证流程

```

    ↓
解析参数
    ↓
✅ Check 1: UTF-8 编码验证
    ↓
✅ Check 2: 不可打印字符比例 (≤2%)
```

**问题**:

**防护**:

```
💡 Common causes:
• Character encoding mismatch (UTF-8 vs GBK/GB2312)
• Copy-paste errors from corrupted sources
• Binary data mixed with text

 Please re-generate the content with correct encoding.
```

**结果**: ✅ 阻止乱码写入，LLM 收到错误反馈后会重新生成


**结果**: ✅ 检测到编码错误，阻止写入

## 🎓 设计决策

### 为什么选择 2% 阈值？

**分析**:
- 正常源代码: 0-1% 不可打印字符（主要是 `\n`, `\t`）
- 包含特殊字符的代码: 1-2%（如 emoji、特殊符号）
- 乱码/损坏文件: 5-50% 不可打印字符

**结论**: 2% 是一个安全的阈值，不会误判正常代码，但能有效捕捉乱码

### 为什么不直接拒绝所有非 ASCII？

**原因**:
1. 现代代码经常包含 Unicode（中文注释、国际化）
2. Rust/Python/JS 都原生支持 UTF-8
3. 拒绝非 ASCII 会严重限制 LLM 的能力

**方案**: 只验证编码有效性，不限制字符集

### 为什么在工具层验证而非 Agent 层？

**原因**:
1. **防御纵深**: 工具层是最后一道防线
2. **简单可靠**: 不需要修改复杂的 Agent 逻辑
3. **独立验证**: 即使 Agent 逻辑有 bug，工具层仍能保护
4. **错误反馈**: 工具层可以直接返回错误给 LLM
