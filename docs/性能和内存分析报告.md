> 分析日期: 2026-04-27  
> 分析范围: crates/ox-core, crates/ox-cli  
> 版本: v0.1.0

## 一、🔴 高优先级问题

### 1. SSE 解析中的字符串频繁复制

**位置**: `crates/ox-core/src/llm/openai.rs:99-104`, `crates/ox-core/src/llm/anthropic.rs:107-112`

```rust
// openai.rs
while let Some(pos) = buffer.find('\n') {
    let line = buffer[..pos].trim_end_matches('\r').to_string(); // ← 每次分配
    buffer = buffer[pos + 1..].to_string();                       // ← 每次分配
```

**问题描述**:  
每个 SSE 数据块都要创建两个新的 `String`，`buffer` 也整体重新分配。在长流式响应中可能有数百上千个 chunk，产生大量短生命周期分配，给 GC（Rust 的 drop 系统）带来压力。

**影响**:  
- 每次 chunk 至少 2 次 `String` 分配
- 一次典型流式响应可能产生 500-2000 次分配
- 增加 CPU 开销和内存碎片

**建议修复**:
```rust
// 方案 1: 使用 drain 原地删除，避免重新分配
while let Some(pos) = buffer.find('\n') {
    let line = buffer[..pos].trim_end_matches('\r');
    // 处理 line (作为 &str)
    buffer.drain(..=pos);  // 原地删除已处理部分
}

// 方案 2: 使用 bytes::BytesMut 做零拷贝解析
```

### 2. `messages.clone()` 在每个 iteration 复制全量消息

**位置**: `crates/ox-core/src/agent/mod.rs:99`

```rust
let msgs = messages.clone();  // ← 每轮迭代完整克隆所有消息
```

**问题描述**:  
在多轮 tool-use 循环中，`messages` 会不断增长（system + history + 工具结果），每次 clone 都深拷贝所有 `Message`（包含大量 String 内容）。

**影响**:  
- 假设 messages 有 50 条、每条平均 500 字符，一次 clone 就是 ~25KB
- 多次 tool-use 循环叠加后更严重
- 在 LLM 流式响应前阻塞等待 clone 完成

**建议修复**:
```rust
// 方案 1: 传入引用而非 clone
let provider_clone = Arc::clone(&provider);
// tokio::spawn(async move {
//     provider_clone.stream_chat(&messages, &schemas, llm_tx).await;
// });

// 方案 2: 使用 Arc<[Message]> 做写时复制（CoW）共享
let messages: Arc<[Message]> = messages.into();
```

### 3. `Message` 全部使用 `#[derive(Clone)]` 导致深度拷贝

**位置**: `crates/ox-core/src/message/mod.rs:10`

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum Message { ... }
```

**问题描述**:  
整个项目中 `Message` 被频繁 `clone()`（agent loop, session persistence, context builder），而 `Message` 内部包含大量 `String` 字段。

**影响**:  
- 每次 clone 都深拷贝所有文本内容
- 在 context builder 中迭代 history 时大量 clone
- 序列化/反序列化时再次拷贝

**建议修复**:
```rust
// 方案: 将大文本字段改用 Arc<str>
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum Message {
    System { content: Arc<str> },
    User { content: Arc<str> },
    Assistant {
        content: Arc<str>,
        #[serde(skip_serializing_if = "Vec::is_empty", default)]
        tool_calls: Vec<ToolCall>,
    },
    ToolResult {
        tool_call_id: String,
        content: Arc<str>,
    },
}
```

## 二、🟡 中优先级问题

### 4. `code_search` 使用阻塞 I/O 在 async 上下文中

**位置**: `crates/ox-core/src/tools/code_search.rs:94`

```rust
let content = match fs::read_to_string(&path) { ... };  // ← 阻塞调用
```

**问题描述**:  
`std::fs::read_to_string` 是阻塞调用，在 Tokio async 上下文中执行会阻塞工作线程。如果搜索大文件或大量文件，会影响整个事件循环。

**影响**:  
- 搜索大文件时 UI 可能出现卡顿
- 阻塞 Tokio 工作线程，影响其他异步任务

**建议修复**:
```rust
// 方案 1: 使用 tokio::fs
let content = match tokio::fs::read_to_string(&path).await { ... };

// 方案 2: 使用 spawn_blocking 包装
let content = tokio::task::spawn_blocking({
    let path = path.clone();
    move || std::fs::read_to_string(&path)
}).await??;
```

### 5. `Session::load()` 一次性读取全部 JSONL 到内存

**位置**: `crates/ox-core/src/message/session.rs:68`

```rust
let lines: Vec<String> = reader.lines().collect::<Result<_, _>>()?;
```

**问题描述**:  
对于长会话，所有行一次性加载到 `Vec<String>`，同时又在第 79-103 行再次 `serde_json::from_str` 解析每条消息，内存使用翻倍。如果 JSONL 文件有 10MB，这步就要额外 ~10MB+ 内存。

**影响**:  
- 启动/恢复会话时内存峰值翻倍
- 长会话（如 50MB+ JSONL）可能占用 >100MB 内存

**建议修复**:
```rust
// 方案: 使用逐行处理迭代器，不一次性 collect
for line_result in reader.lines() {
    let line = line_result?;
    // 立即解析并处理，不保留原始字符串
    if let Ok(msg) = serde_json::from_str::<Message>(&line) {
        messages.push(msg);
    }
}
```

### 6. `get_rendered_lines` 每帧 flatten 全部缓存

**位置**: `crates/ox-cli/src/terminal/output_pane.rs:186-192`

```rust
let mut result = Vec::with_capacity(self.lines.len() * 2);
for entry in &self.rendered_cache {
    if let Some(lines) = entry {
        result.extend(lines.iter().cloned());
    }
}
result  // ← 每帧重新构建完整 Vec
```

**问题描述**:  
当 `lines` 接近 `MAX_LINES=2000` 时，每帧都要 clone 数千个 `Line<'static>`。虽然有渲染缓存避免重复解析 Markdown/语法高亮，但 flatten 操作本身仍有开销。

**影响**:  
- 每帧 33ms 内执行大量 clone 操作
- 终端窗口大时更明显（需要更多 Line 对象）

**建议修复**:  
- 只 flatten 可见窗口范围 `visible_start..visible_end`
- 不可见行的缓存保留但不 clone
- **详细方案见下方"方案一"**

---

### 7. `append_message` 每次都打开/关闭文件

**位置**: `crates/ox-core/src/message/session.rs:132-137`

```rust
let mut file = OpenOptions::new()
    .create(true)
    .append(true)
    .open(&self.file_path)?;  // ← 每次 append 都重新 open
writeln!(file, "{json}")?;
file.flush()?;
```

**问题描述**:  
每条消息都重新打开文件，涉及系统调用开销。在 agent 一轮对话中可能追加 10+ 条消息。

**影响**:  
- 每次 open/close 涉及系统调用
- 高频率写入时性能下降

**建议修复**:
```rust
// 方案: 在 Session 结构体中保持 File 句柄
pub struct Session {
    pub meta: SessionMeta,
    pub messages: Vec<Message>,
    file_path: PathBuf,
    file_handle: Option<File>,  // ← 保持打开
}
```

### 8. `CostTracker::save()` 同步写磁盘

**位置**: `crates/ox-core/src/cost/mod.rs:138-145`

```rust
fn save(&self) -> anyhow::Result<()> {
    let json = serde_json::to_string_pretty(self)?;
    std::fs::write(&self.file_path, json)?;  // ← 同步阻塞写
}
```

**问题描述**:  
这个方法在 `record()` 中被调用（`cost/mod.rs:94`），发生在 agent 主循环的 `TurnDone` 处理路径上，会短暂阻塞 UI 线程。

**影响**:  
- UI 帧延迟（磁盘 I/O 阻塞）
- 在慢速磁盘上更明显

**建议修复**:  
- 使用异步写入 (`tokio::fs::write`)
- 或延迟批量写入（累积多次记录后一次写入）

## 三、🟢 低优先级问题

### 9. `ToolRegistry::schemas()` 每次都重新构建

**位置**: `crates/ox-core/src/tools/mod.rs:125-134`

每次 `run_agent_turn` 都调用 `tool_registry.schemas()` 创建新的 `Vec<ToolSchema>`（包含 `serde_json::Value` 克隆）。工具 schemas 是静态不变的，可以缓存或返回引用。

**影响**: 小到中等（工具数量少，通常 <15 个）

---

### 10. `rendered_cache` 使用 `Line<'static>`

**位置**: `crates/ox-cli/src/terminal/output_pane.rs:36`

```rust
rendered_cache: Vec<Option<Vec<Line<'static>>>>,
```

`'static` 意味着每个 `Span` 中的文本都必须是 `String`（拥有所有权），不能借用。这导致每次都要 clone 字符串到缓存中。如果能缩短生命周期到 `'a`，可以在渲染时借用原始数据。不过 ratatui 的 API 设计使得 `'static` 是常见的做法，改动收益有限。

---

### 11. 字符串截断的重复逻辑

在 `agent/mod.rs`（行 271-274, 370-374）和 `main.rs`（行 283-288）等多处重复实现"截断 200 字符"逻辑，可以抽取为工具函数。

## 四、📊 内存泄漏风险排查

| 风险点 | 状态 | 说明 |
|--------|------|------|
| `OutputPane` 无限制增长 | ✅ 已防护 | `MAX_LINES=2000` + `trim_excess()` |
| `messages` Vec 无限增长 | ⚠️ 部分防护 | `ContextBuilder` 有 token budget 裁剪，但 `history` Vec 本身不裁剪 |
| `rendered_cache` 膨胀 | ✅ 已防护 | 与 `lines` 同步 trim |
| Channel 消息积压 | ⚠️ 潜在风险 | `UnboundedChannel` 无背压，agent 快速产生事件时可能堆积 |

### `history` 持续增长问题

`history` Vec 在 `main.rs:534-536` 和 `main.rs:304` 只增不减。虽然传给 LLM 的 `turn_messages` 会被 `ContextBuilder` 裁剪，但 `history` 本身会一直增长。长会话中这可能累积大量内存。

**建议**: 当 `history` 超过一定阈值时，裁剪旧消息（保留最近的 N 条），或定期压缩为摘要。

| 优先级 | 问题 | 预计收益 | 实现难度 |

## 八、🔧 具体实施方案

### 方案一: 可见行 clone 优化（解决 #6）

**目标**: 帧渲染从 O(total_lines) clone 降至 O(visible_lines) clone

**改动**:

1. `output_pane.rs` — `get_rendered_lines` 增加可见范围参数:

```rust
pub fn get_rendered_lines(
    &mut self,
    output_width: usize,
    visible_start: usize,  // 新增
    visible_count: usize,  // 新增
    mut render_fn: impl FnMut(&OutputLine, usize) -> Vec<Line<'static>>,
) -> (Vec<Line<'static>>, usize)  // 返回 (visible_lines, total_lines)
```

逻辑:
- 仍对全部行做缓存填充（确保 `rendered_cache[i]` 有值）
- 但 flatten 时只遍历 `visible_start..visible_start+visible_count`
- 返回可见行 + 总行数（用于 scroll 计算）

2. `render.rs` — 两步渲染:

```rust
// 第一步: 计算总行数（不 clone）
let total_lines = app.output.total_rendered_line_count();

// 第二步: 计算可见范围
let max_scroll = total_lines.saturating_sub(inner_height);
let effective_scroll = max_scroll.saturating_sub(scroll_offset as usize);
let visible_start = total_lines.saturating_sub(inner_height + effective_scroll);
let visible_count = inner_height.min(total_lines);

// 第三步: 只 clone 可见行
let lines = out.get_rendered_lines(output_width, visible_start, visible_count, |ol, w| {
    render_single_line(ol, w, md)
});
```

**预期**: 2000 行对话帧耗时从 ~25ms 降至 ~3ms

---

### 方案二: 消除 history 与 session.messages 双重存储

**目标**: 节省 ~10MB 内存（50 轮对话）

**改动**:
1. `main.rs` — 移除 `let mut history: Vec<Message>`，所有引用改为 `&session.messages`
2. `ContextBuilder::build()` — 传 `&session.messages` 而非 `&history`
3. `TurnDone` — 不再 `history.extend()`，已通过 `session.append_message()` 持久化
4. `SlashCommand::Resume` — 不再 `*history = ...clone()`
5. `SlashCommand::Debug` — `history.len()` → `session.messages.len()`

**风险**: `session.messages` 需要 `&Session` 或 `&mut Session` 访问，需确保与 `append_message` 不同时借用。当前代码中 `session` 只在 TurnDone / UserInput 时修改，渲染时不访问，无冲突。

### 方案三: SSE buffer 原地解析（解决 #1）

**目标**: 减少 30-50% 流式解析内存分配

**改动** (以 openai.rs 为例):

```rust
// Before:
while let Some(pos) = buffer.find('\n') {
    let line = buffer[..pos].trim_end_matches('\r').to_string();
    buffer = buffer[pos + 1..].to_string();
    // process line
}

// After:
while let Some(pos) = buffer.find('\n') {
    let line_end = buffer[..pos].trim_end_matches('\r').len();
    let line = buffer[..line_end].to_string();  // 只 clone 行内容
    buffer.drain(..=pos);  // 原地删除，不重新分配整个 buffer
    // process line
}
```

### 方案四: Session 持有 File 句柄（解决 #7）

**目标**: 消除每次 append_message 的 open/close 开销

**改动**:

```rust
pub struct Session {
    pub meta: SessionMeta,
    pub messages: Vec<Message>,
    file_path: PathBuf,
    file_handle: Option<BufWriter<File>>,  // 新增
}

impl Session {
    pub fn append_message(&mut self, msg: Message) -> anyhow::Result<()> {
        // ...
        if let Some(ref mut file) = self.file_handle {
            writeln!(file, "{json}")?;
            // 不 flush 每次，依赖 BufWriter 缓冲
        }
    }
}
```
